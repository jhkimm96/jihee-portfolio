# publish-quality 스킬 설계

- 날짜: 2026-07-20
- 상태: 설계 승인됨 (구현 전)
- 산출물: ① 사용자 스킬 `publish-quality` ② 포트폴리오 `quality` 콘텐츠 타입 + 대시보드 배선

## 배경 / 목적

코드베이스(우선 대상: beadv6_6_3JMT_BE의 각 서비스)를 **같은 잣대로 반복 점검**해서
리팩토링 대상(중복 로직, 미사용 코드, 비대한 클래스, 레이어 위반, 서비스 간 통신 문제 등)을
찾고, 그 결과를 포트폴리오에 **날짜별 정량 스냅샷**으로 쌓아 "품질이 좋아졌다"를
**추세 그래프 대시보드**로 보여준다.

포트폴리오 기록의 역할 구도는 두 축이다:

- **quality** (이 설계): 정량 스냅샷 + 추세 그래프
- **decision** (기존 publish-decision): 개별 설계 결정의 근거(ADR) — 품질 점검 중
  나온 설계 문답이 여기로 흘러들어간다

기존 reviews 타입·publish-reviews 스킬은 이 설계와 무관하며 손대지 않는다(그대로 방치).

## 2. 분석 파이프라인

### 2-1. 스크립트 메트릭 (스킬 `scripts/`에 번들, 매 회차 동일 기준)

| 메트릭 | 도구/방법 | 고정 기준 (v1) |
|---|---|---|
| 중복 블록 수 · 중복률(%) | jscpd (npx) | `--min-tokens 70 --min-lines 5`, 대상: 스코프의 `src/main/java` |
| 비대 클래스 수 | 자체 스캔 스크립트 | 300 LOC 초과 **또는** 메서드 15개 초과 |
| 전체 LOC · 파일 수 | 자체 스캔 스크립트 | 산정 방식(주석·공백 처리)은 스크립트에 고정 |

- 테스트 코드(`src/test`)는 메트릭 대상에서 제외한다 (본문 품질 측정이 목적).
- 스크립트가 실패하면(jscpd 실행 불가 등) 해당 메트릭 없이 진행하지 않는다 —
  회차 간 비교성이 깨지므로 **중단하고 원인을 보고**한다.

### 2-2. Claude 루브릭 (고정 체크리스트, 11개 카테고리)

각 위반은 `카테고리 / 심각도 / 파일:라인 / 근거` 로 기록한다.

| 카테고리 | 판정 내용 |
|---|---|
| `controller-thin` | 컨트롤러에는 호출·위임 로직만. 비즈니스 로직 혼입 금지 |
| `entity-encapsulation` | 엔티티는 setter 반복 호출이 아니라 자기 상태를 스스로 관리. 단, 엔티티에 과도한 로직 집중도 위반 |
| `http-semantics` | HTTP 상태코드를 의미에 맞게 반환 |
| `logging-quality` | 원인 추적에 도움 되는 좋은 로그 (필요한 곳에 있고, 무의미한 남발이 없음) |
| `exception-discipline` | try-catch는 실제 예외가 날 만한 곳에만. 무분별한 감싸기 금지 |
| `layer-separation` | 외부 기술(인프라 클라이언트, SDK 등)을 비즈니스 서비스에 직접 혼입 금지 |
| `dead-code` | 실질적 미사용 코드. **미사용 외부 연동 코드 포함** — 호출되지 않는 클라이언트, 죽은 토픽의 컨슈머, 쓰지 않는 외부 서비스 설정·의존성 잔재 |
| `duplication-semantic` | 스크립트(jscpd)가 못 잡는 의미론적 중복 로직 |
| `diagnosability` | 예외 삼킴, 모호한 에러 메시지 등 오류 원인 즉시 추적을 방해하는 코드 |
| `integration-robustness` | 다른 서비스와의 통신(gRPC·Kafka·REST)이 실패를 제대로 다루는지 — 타임아웃 없는 원격 호출, 실패·에러 응답 무시, 컨슈머 예외 삼킴, 응답 무검증 |
| `service-boundary` | 서비스 경계 위반 — 다른 서비스의 DB/스키마 직접 접근, 다른 서비스 도메인 로직 중복 구현, 순환 호출 유발, 이벤트가 자연스러운 곳의 동기 호출 체인 |

**판정 원칙** (감점 항목이 아니라 루브릭이 지켜야 할 규칙):

- 모든 코드를 억지로 테스트할 필요는 없다 — 테스트 커버리지 부족 자체를 위반으로 지적하지 않는다.
- 위반은 실제 파일:라인을 확인한 것만 기록한다. 추측 금지.
- 서비스 단위 스냅샷이므로 통신·경계 판정은 **그 서비스 쪽 코드만** 대상 (상대 서비스는 검사 안 함 — 본문 "한계"에 명시).
- 회차 간 일관성을 위해 루브릭 문구·심각도 정의를 스킬에 고정하고, 실행 시 임의로 기준을 바꾸지 않는다.

**심각도 정의 (고정)**:

- **High**: 동작·운영에 실제 리스크 (예: 예외 삼킴으로 원인 소실, 원격 호출 실패 무시로 데이터 불일치 가능, 다른 서비스 DB 직접 조작)
- **Medium**: 유지보수성을 분명히 해침 (예: setter 남발, 서비스에 인프라 SDK 직접 사용, 타임아웃·재시도 미설정, 도메인 로직 중복 구현)
- **Low**: 사소한 개선 여지 (예: 로그 레벨 부적절, 불필요한 try 블록, 장황한 연동 코드)

## 3. 설계 질문 단계 (비점수)

분석 중 "보통 이런 상황엔 X 패턴을 쓰는데 여긴 다르게 했다" 싶은 지점을 수집해뒀다가,
결과 제시 후 **하나씩** 질문한다. 예: "주문 취소에서 재고 복구를 동기 gRPC로 호출하는데,
실패하면 취소가 롤백되나요? 보통 이벤트로 푸는데 의도가 있었나요?"

- **실제로 더 나은 대안이 알려진 지점만** 질문한다. 트집용·스타일 취향 질문 금지.
- 문답 결과는 점수에 반영하지 않는다. 루브릭 위반으로 이미 잡힌 항목은 사용자가 의도한
  결정이라고 답해도 **점수에서 빼지 않는다** (빼기 시작하면 회차 간 비교가 조작 가능해짐).
  대신 해당 발견에 "의도된 결정 → decision 링크"를 달아 맥락을 제공한다.
- 기록할 가치가 있는 결정이 나오면 **기존 publish-decision 스킬로 게시를 제안**한다
  (그 스킬의 게이트·승인 절차를 그대로 따름). 별도 새 스킬은 만들지 않는다.
- 질문에 답하지 않고 넘어가도 게시는 진행 가능하다.

## 4. 종합 점수 산식 (v1, 고정)

```
score = max(0, 100
  − 5 × High 건수
  − 2 × Medium 건수
  − 0.5 × Low 건수
  − 1 × 비대 클래스 수
  − 1 × max(0, ceil(중복률% − 3)))   # 중복률 3% 초과분부터 1%p당 1점
```

- frontmatter에 `formulaVersion: 1`을 기록한다. 산식·가중치·스크립트 고정 기준(§2-1)을
  바꾸면 버전을 올리고, 대시보드는 버전이 다른 스냅샷 비교 시 주석을 표시한다.
- 점수는 소수 첫째 자리까지.

## 5. 콘텐츠 스키마 (`quality` 타입)

- 경로: `content/{project}/quality/{scope}-{YYYY-MM-DD}.mdx`
  (예: `content/prompthub/quality/product-service-2026-07-20.mdx`)
- 날짜별 스냅샷 누적. 같은 경로 존재 시 자동 덮어쓰기 금지, 사용자 확인 후 구분자 추가.
- `date`는 게시 시점의 실제 오늘 날짜.

### frontmatter (`qualityFrontmatterSchema`)

```yaml
title: "product-service 품질 스냅샷 (2026-07-20)"   # 한글
date: "2026-07-20"
scope: "product-service"          # 서비스 slug — 대시보드 추세선 그룹 키
score: 78.5
formulaVersion: 1
metrics:
  locTotal: 12345
  files: 180
  duplicationBlocks: 14
  duplicationPct: 4.2
  oversizedClasses: 3
findings:                          # 카테고리별 건수만 — 상세는 본문
  - { category: "controller-thin", high: 1, medium: 2, low: 0 }
  # ... 11개 카테고리 전부 기록. 위반 없으면 0 (누락 아님 — 추세 비교용)
summary: "1~2문장 총평"
tags: ["code-quality", "refactoring"]
draft: false
```

### 본문 템플릿 (고정)

```
## 범위
어느 서비스, 어느 커밋/브랜치 기준, 무엇을 봤는지.

## 요약
총평 한 단락. 이전 스냅샷이 있으면 점수 변화 언급.

## 카테고리별 발견
카테고리마다: 위반 목록 (심각도 · 파일:라인 · 근거 · 권장 수정 방향).
의도된 결정으로 확인된 발견은 decision 링크 표기 (§3).

## 개선된 항목
이전 스냅샷 대비 해소된 위반. 같은 프로젝트의 troubleshooting/decision으로
해결·결정된 건은 링크 연결 — 같은 {project}의 글만, 명백한 매칭만, 애매하면 사용자 확인.

## 한계
안 본 것 명시. 정적 스냅샷임과 서비스 단위 한계(상대 서비스 미검사) 명시.
```

## 6. 포트폴리오 대시보드 배선 (1회 선행 작업)

스킬이 아니라 **이번 구현 프로젝트에서 포트폴리오 저장소에 직접** 작업한다.
스킬 자체는 포트폴리오 코드를 절대 수정하지 않으며, 배선이 없으면 사용자에게 알리고 중단한다.

- velite: `quality` 컬렉션 + `qualityFrontmatterSchema` (`content/schemas.ts`)
- 라우트: `app/quality/page.tsx` (대시보드), `app/quality/[...slug]/page.tsx` (스냅샷 상세)
- `lib/content-data`에 quality 접근자 추가
- 대시보드 구성:
  - scope(서비스) 셀렉터
  - 헤드라인: 선택된 scope의 **종합 점수 추세선**
  - 상세: 카테고리별 위반 건수 추세, 메트릭(중복률·비대 클래스) 추세
  - 최신 스냅샷 요약 카드 (→ 상세 페이지 링크)
  - 스냅샷이 1개면 추세선 대신 단일 점 표시 (정상 동작)
- 차트 라이브러리: 포트폴리오 기존 의존성 확인 후 구현 계획에서 결정 (신규 의존성 최소화 우선)

## 7. 게이트 · 컨벤션

1. **내용 승인 게이트**: 분석 결과 초안 전체(frontmatter + 본문)를 보여주고 명시적 승인 후에만
   파일 작성. 승인 전에 어떤 파일도 쓰지 않는다.
2. **민감정보 게이트** (승인과 별개 단계, 생략 불가): 회사/조직 실명 · 내부 도메인/URL ·
   자격증명/키 · 팀원 실명 · 고객사/미공개 사업 정보 · 미조치 취약점 상세 — 앞 다섯 항목은
   명시적 "없음" 답변으로만 통과. 미조치 취약점은 일반화하거나 제외하거나 `draft: true`.
3. **빌드 검증**: `npm run build` 성공 시에만 커밋. 실패 시 파일은 워킹트리에 두고 에러 보고.
4. **커밋**: portfolio repo 컨벤션 `<카테고리>: 한글설명` 형식 (예: `품질: product-service 품질 스냅샷 추가`).
   `git push` 금지.

## 8. 에러 처리

- 스크립트 메트릭 실패 → 중단 + 원인 보고 (부분 진행 금지, §2-1)
- 배선 미존재 → 중단 + 배선 필요 안내 (§6)
- slug 충돌 → 사용자 확인 (§5)
- 이전 스냅샷 없음 → 첫 스냅샷으로 정상 진행 (그래프에 점 하나)

## 9. 구현 순서

1. **포트폴리오 배선**: schemas.ts + velite 컬렉션 + 라우트 + 대시보드 (스냅샷 fixture로 빌드 확인)
2. **스킬 작성**: skill-creator로 SKILL.md + scripts/ 작성, product-service 대상 실제 실행으로 검증
3. 첫 실제 스냅샷 게시로 엔드투엔드 확인

## 비범위 (Out of scope)

- CI 연동·자동 실행 (스킬은 명시적 요청시에만)
- BE 레포 코드 수정 (분석만 하고 리팩토링 실행은 별도 요청)
- 다른 언어/프레임워크 지원 (v1은 Java/Spring 서비스 기준. 스크립트 기준값이 Java 전제)
- 포트폴리오 push (항상 사용자가 직접)
- 기존 reviews 타입·publish-reviews 스킬의 변경·삭제 (그대로 둠. 정리는 별도 요청 시)
