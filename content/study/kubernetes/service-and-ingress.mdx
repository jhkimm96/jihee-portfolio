## 왜 Service가 필요한가 — Pod IP는 계속 바뀐다

공식 문서가 문제를 이렇게 짚는다:

> "Pod는 임시적 자원이다... 백엔드 역할을 하는 Pod들이 있을 때, 프론트엔드는 어떤 IP 주소로 연결해야 하는지 어떻게 알아낼 수 있을까?"

Pod는 재시작될 때마다 IP가 바뀔 수 있다. **비유**: 계속 이사 다니는 친구한테 편지를 보내려고 매번 "지금 어디 사는지" 새로 알아내는 건 너무 번거롭다. 대신 "우체국 사서함 번호" 하나만 알면, 그 친구가 어디로 이사 가든 사서함으로 보낸 편지는 알아서 전달된다 — 이 사서함 번호 역할을 하는 게 **Service**다.

## Service란 — 고정된 전화번호

> "Service는 클러스터에서 하나 이상의 Pod로 실행 중인 네트워크 애플리케이션을 노출하는 방법이다"

가장 기본 타입인 **ClusterIP**는 "고정된 가상 IP 하나"를 배정받고, `selector`로 지정한 라벨이 붙은 Pod들한테 트래픽을 나눠서 보내준다. PromptHub의 `product-service` Service를 보면:

```yaml
kind: Service
spec:
  type: ClusterIP
  selector:
    app.kubernetes.io/name: product-service
  ports:
    - name: http
      port: 8082
      targetPort: http
    - name: grpc
      port: 9082
      targetPort: grpc
```

`selector`가 "`app.kubernetes.io/name: product-service` 라벨이 붙은 Pod한테 트래픽 보내"라는 뜻이다. 그래서 다른 서비스들은 실제 Pod IP를 몰라도, `product-service:8082`라는 **이름**만 알면 항상 지금 살아있는 Pod로 연결된다.

## headless Service — 특별한 버전

3편에서 postgres StatefulSet에 `serviceName: postgres-headless`가 있었다. 이게 headless(비머리) Service다.

| | 일반 ClusterIP Service | headless Service |
|---|---|---|
| IP | 고정 가상 IP 하나 배정 | `clusterIP: None` — 배정 안 함 |
| DNS로 조회하면 | Service의 고정 IP 하나 나옴 | **Pod들 각자의 실제 IP**가 그대로 나옴 |
| 용도 | "아무 Pod나 상관없이 트래픽만 도착하면 됨" | "이 특정 Pod한테 정확히 연결해야 함" |

PromptHub는 postgres용으로 **둘 다** 갖고 있다:

```yaml
# service.yaml — 일반 앱들이 DB 접속할 때 씀
spec:
  type: ClusterIP
  selector:
    app.kubernetes.io/name: postgres

# headless-service.yaml — StatefulSet 자신의 신원 관리용
spec:
  clusterIP: None
  selector:
    app.kubernetes.io/name: postgres
```

각 서비스 Pod들은 그냥 `postgres:5432`(일반 Service)로 접속하면 되고, StatefulSet 내부적으로 "Pod가 순서대로 자기 신원을 유지"하는 데는 headless Service가 쓰인다. 공식 문서 표현으로 "StatefulSet의 각 Pod는 안정적인 네트워크 식별자가 필요하며, headless Service를 통해 Pod 이름으로 직접 접근할 수 있다"는 게 이 이유다.

<summary><b>🔍 headless Service가 실제로 어떤 이점이 있는지 더 자세히</b></summary>

핵심은 "아무나 연결돼도 되는가" vs "정확히 이 애랑 연결해야 하는가"다.

**일반 ClusterIP Service** = 고객센터 대표번호. 전화 걸면 지금 한가한 상담원 아무나 랜덤으로 연결된다. "영업시간이 몇 시예요?" 같은, 누가 받든 상관없는 질문엔 이게 딱이다.

**headless Service** = 직통 번호. 각 사람마다 고유 번호가 있어서 "김대리님 바꿔주세요"처럼 **특정 한 명을 콕 집어서** 연결할 수 있다.

**DB 클러스터에서 왜 "콕 집기"가 필요한가**: 예를 들어 PostgreSQL을 3대로 복제 클러스터를 짰다고 하자 — 0번은 대표(primary, 쓰기 담당), 1번·2번은 복제본(replica, 읽기 전용)이다. 셋 다 일반 Service 뒤에 있으면, 앱이 접속할 때마다 Kubernetes가 랜덤으로 0/1/2번 중 아무 데나 연결해준다. 쓰기 요청이 하필 1번(복제본)한테 갔다면 실패한다 — 복제본은 쓰기를 못 받기 때문이다. "쓰기는 무조건 0번한테"라는 걸 강제할 방법이 일반 Service엔 없다.

headless Service를 쓰면 DNS가 `postgres-0.postgres-headless`, `postgres-1.postgres-headless`처럼 각자 고유 이름을 준다. 그래서 "쓰기는 무조건 `postgres-0`으로"처럼 특정 멤버를 정확히 지목할 수 있다.

**솔직히 말하면**: PromptHub의 PostgreSQL은 지금 `replicas: 1`이라, Pod가 하나뿐이면 "아무나 연결"이든 "콕 집어 연결"이든 결과가 똑같아서 이 이점이 지금 당장 눈에 보이지는 않는다. 이건 나중에 여러 대로 늘릴 때를 대비한 밑준비에 가깝다 — StatefulSet은 "언젠가 여러 개로 늘어날 수 있다"는 전제로 설계된 오브젝트라, 처음부터 headless Service를 같이 두는 게 관례다.

## Ingress란 — 외부에서 들어오는 정문

지금까지 본 Service는 전부 **클러스터 안에서만** 통하는 주소다. 외부 인터넷에서 들어오는 요청은 다른 게 받는다:

> "Ingress는 클러스터 외부에서 클러스터 내 서비스로의 접근을 관리하는 API 객체다, 보통 HTTP" ... "Ingress를 만들었다고 저절로 뭔가 되는 게 아니다 — 반드시 Ingress Controller가 있어야 한다"

PromptHub의 `gateway-ingress.yaml` 전체다:

```yaml
kind: Ingress
spec:
  ingressClassName: nginx
  rules:
    - http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: apigateway
                port:
                  number: 8000
```

"`/`로 들어오는 요청은 전부 `apigateway`라는 Service의 8000번 포트로 보내라"는 규칙 딱 하나다.

## 정리 — 요청이 실제로 어떻게 흘러가나

- **Ingress** = 정문 (외부 → 클러스터 진입점, 딱 하나)
- **Service** = 클러스터 안에서 "이름으로 Pod 찾아가는 방법" (여러 개, 서비스마다 하나씩)

PromptHub 흐름: 외부 요청 → Ingress → `apigateway` Service(ClusterIP) → apigateway Pod → (Eureka로 실제 서비스 위치 조회) → 각 서비스의 Service → 그 서비스 Pod
