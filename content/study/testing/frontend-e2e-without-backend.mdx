## 배경

프론트엔드 변경을 "제대로" 검증하려면 타입체크·유닛테스트로는 부족하다. 실제로 앱을
띄우고, 사용자가 하듯 클릭해서, 화면이 바뀌는 걸 눈으로 봐야 한다. 문제는 검증하려는
화면이 **로그인 + 특정 권한(판매자)** 뒤에 있고, 동작이 **백엔드(파일 업로드용 S3
presigned URL 발급 등)** 에 의존할 때다. 계정도 없고 백엔드도 못 띄우는 상황에서
어떻게 혼자 end-to-end로 확인하나?

핵심 아이디어는 하나다. **진짜 브라우저를 코드로 조종하되, 인증과 백엔드만 가짜로
채워넣는다.** 검증 대상인 UI 로직(유형별 폼 분기, 업로드 흐름)은 진짜로 실행시키고,
그 주변의 인증·네트워크 경계만 통제한다.

<details>
<summary><b>🔍 Playwright가 뭔데?</b></summary>

브라우저(크롬 등)를 **코드로 자동 조작**하는 도구다. 사람이 마우스로 클릭·입력·스크롤
하던 걸 `page.click()`, `page.setInputFiles()` 같은 명령으로 시킨다. 화면을 그대로
띄운 채(headless면 화면 없이 백그라운드로) 실제 렌더링·자바스크립트 실행이 일어나므로,
"진짜 앱을 사용자처럼 굴린다"에 가깝다. 스크린샷·네트워크 가로채기 기능도 내장돼 있어
E2E 검증에 쓴다.

</details>

## 1. 인증 게이트 통과 — 쿠키 + localStorage 주입

이 앱은 인증을 두 겹으로 막는다.

- **미들웨어(서버)**: 요청 쿠키의 `token`·`roles`를 보고 판매자 전용 경로를 가드
- **클라이언트 스토어**: zustand `persist`가 localStorage에 로그인 상태를 저장

<details>
<summary><b>🔍 "클라이언트 스토어"가 뭐야?</b></summary>

브라우저(=클라이언트) 안에 살아있는 **전역 상태 저장소**다. 로그인한 유저 정보, 장바구니
같은 걸 여러 컴포넌트가 공유해야 하는데, 이걸 한곳에 모아두는 게 스토어다. 이 앱은
`zustand`라는 라이브러리로 만든다. "서버 DB"가 아니라 "지금 이 브라우저 탭의 메모리"에
있는 상태라고 보면 된다.

</details>

<details>
<summary><b>🔍 zustand persist는 뭐고, 왜 localStorage에 저장해?</b></summary>

스토어는 기본적으로 메모리에 있어서 **새로고침하면 사라진다**. 로그인 상태가 새로고침마다
날아가면 곤란하다. `persist`는 zustand의 미들웨어로, 스토어 값이 바뀔 때마다 자동으로
`localStorage`(브라우저에 영구 저장되는 키-값 공간)에 복사해두고, 다음에 페이지가 열리면
거기서 다시 읽어와 복원한다. 그래서 새로고침해도 로그인이 유지된다.

우리는 이 "복원" 지점을 역이용한다. 앱이 읽어갈 localStorage에 **미리 로그인된 값을
심어두면**, 앱은 "아, 이미 로그인돼 있네" 하고 믿는다.

</details>

<details>
<summary><b>🔍 zustand persist 저장 포맷</b></summary>

persist는 아무 모양으로나 저장하지 않고 정해진 껍데기를 쓴다:

```json
{ "state": { "...실제 상태..." }, "version": 0 }
```

`state` 안에 스토어 값이 들어가고, `version`은 스키마 마이그레이션용 번호다. 그래서
주입할 때도 이 포맷을 **그대로** 맞춰야 앱이 정상적으로 읽는다. `{ user, token,
isLoggedIn }`만 달랑 넣으면 persist가 인식 못 한다.

</details>

Playwright는 둘 다 페이지 로드 *전에* 심을 수 있다. 쿠키는 `addCookies`, localStorage는
`addInitScript`로.

```js
await ctx.addCookies([
  { name: 'token', value: 'FAKE_TOKEN', url: BASE },
  { name: 'roles', value: 'seller', url: BASE },
]);
// zustand persist 포맷 그대로: { state: {...}, version: 0 }
await ctx.addInitScript((s) => localStorage.setItem('auth', s), JSON.stringify({
  state: { user: { id: 'u1', roles: ['seller'] }, token: 'FAKE_TOKEN', isLoggedIn: true },
  version: 0,
}));
```

<details>
<summary><b>🔍 addInitScript는 언제 실행돼? (타이밍이 왜 중요한가)</b></summary>

`addInitScript`로 등록한 코드는 **매 페이지의 어떤 스크립트보다도 먼저** 실행된다.
순서가 핵심이다:

1. `addInitScript` → localStorage에 로그인 값 심기 ✅ (여기가 가장 먼저)
2. 그 다음 앱 코드가 로드되며 zustand persist가 localStorage를 읽음
3. 이미 로그인 값이 있으니 앱은 로그인 상태로 시작

만약 페이지가 뜬 *뒤에* 값을 넣으면, 앱은 이미 "비로그인"으로 읽어버린 뒤라 늦는다.
그래서 "로드 전에 심는다"가 이 기법의 필수 조건이다.

</details>

## 2. 백엔드 경계 스텁 — route()로 응답 가로채기

axios가 `/api/v2/*`로 내보내는 요청을 전부 가로채, 필요한 것만 가짜 응답으로 돌려준다.
특히 파일 업로드는 `presign 발급(POST) → S3에 직접 PUT` 2단계라, 둘 다 스텁해야
프론트 흐름이 끝까지 돈다.

<details>
<summary><b>🔍 "스텁(stub)"이 뭐고, "둘 다 스텁한다"는 무슨 말?</b></summary>

**스텁 = 진짜 대신 세워두는 가짜 응답.** 실제 백엔드/S3를 부르는 대신, "이 주소로
요청 오면 이런 답을 주라"고 미리 정해두는 것이다.

파일 업로드는 한 번에 안 끝나고 두 군데를 부른다:

1. **백엔드에 presign 요청** → "이 URL로 파일 올려" 하고 **업로드용 임시 주소**를 받음
2. 받은 그 **S3 주소로 파일을 직접 PUT** 업로드

그래서 "둘 다 스텁"은 ① presign 발급 응답과 ② S3 PUT 응답을 **둘 다** 가짜로
막는다는 뜻이다. 하나만 막으면 흐름이 중간에서 끊겨 끝까지 확인이 안 된다.

</details>

<details>
<summary><b>🔍 route()가 axios 요청을 어떻게 가로채?</b></summary>

`ctx.route(패턴, 핸들러)`는 브라우저가 내보내는 **네트워크 요청을 URL 패턴으로 가로채는**
Playwright 기능이다. axios든 fetch든, 결국 브라우저의 네트워크 계층을 통해 나가는데
Playwright가 그 계층 위에 앉아 있다. 패턴(`**/api/v2/**`)에 걸리면 실제 서버로 안 보내고
핸들러가 `route.fulfill(...)`로 만든 가짜 응답을 대신 돌려준다. axios 입장에선 서버가
정상 응답한 것과 구분이 안 된다 — 그래서 앱 코드를 하나도 안 고치고 백엔드만 갈아끼울 수
있다.

</details>

```js
await ctx.route('**/api/v2/**', async (route) => {
  const url = route.request().url();
  const method = route.request().method();
  if (url.includes('/users/me')) return json(route, { data: { roles: ['seller'] } });
  if (url.includes('/uploads') && method === 'POST')
    return json(route, { data: { uploadUrl: BASE + '/__s3_stub', fileUrl: 'https://s3.example.com/x.pptx' } });
  return json(route, { data: [] }); // 목록성 응답 기본값
});
await ctx.route('**/__s3_stub', (route) => route.fulfill({ status: 200, body: '' })); // S3 PUT
```

## 3. 관찰 — 클릭하고, 캡처하고, payload를 검사한다

검증은 "코드가 맞아 보인다"가 아니라 **실행된 결과를 본다**. 유형 탭을 실제로 클릭해
폼이 바뀌는지 스크린샷으로 남기고, 파일을 넣어 네트워크로 나간 payload를 직접 확인한다.

```js
await page.getByText('PPT', { exact: true }).first().click();
await page.setInputFiles('input[type=file][accept*=".pptx"]', tmpFile);
// → presign POST body: { purpose: 'file', fileName: 'sample.pptx', productType: 'PPT' }
// → S3 PUT 호출됨, 화면에 파일 칩 렌더
```

<details>
<summary><b>🔍 payload가 뭐고, 여기 뭐가 들어있어?</b></summary>

**payload = 요청에 실려 나가는 본문 데이터.** 여기선 프론트가 presign을 요청할 때
"어떤 파일을, 무슨 용도로, 어떤 상품유형으로" 올릴지를 담아 보낸다:

```json
{ "purpose": "file", "fileName": "sample.pptx", "productType": "PPT" }
```

- `purpose: 'file'` — 썸네일/이미지가 아니라 **산출물 파일**임을 표시
- `fileName` — 확장자로 content-type을 정하는 근거
- `productType: 'PPT'` — 내가 PPT 탭을 눌렀다는 게 여기까지 전달됐다는 증거

</details>

<details>
<summary><b>🔍 그래서 이 단계에서 정확히 "뭘" 확인하는 거야?</b></summary>

세 가지를 한 번에 본다:

1. **UI 분기** — PPT 탭을 누르니 폼이 파일 업로드 박스로 바뀌었나 (스크린샷)
2. **배선(wiring)** — 내가 누른 `PPT`가 상태를 거쳐 네트워크 payload의
   `productType: 'PPT'`까지 정확히 흘러갔나
3. **응답 후 반응** — 업로드가 끝나자 화면에 파일 칩이 떴나

즉 "화면 → 상태 → 네트워크 → 다시 화면"이라는 한 바퀴가 끊김 없이 도는지를 본다.
payload만 맞아도, 스크린샷만 맞아도 안 되고 **연결된 전체 흐름**을 확인하는 게 포인트다.

</details>

## 스텁이 틀리면 검증도 틀린다

한 번은 목록 응답 기본값을 `{ data: {} }`(객체)로 뒀더니, 헤더가 알림 목록을
`notifList.filter(...)`로 처리하다 `filter is not a function`으로 **레이아웃째 크래시**
했다. 검증하려던 페이지는 뜨지도 않았다. 배열을 기대하는 자리에 객체를 물린 스텁
탓이었고, `{ data: [] }`로 고치니 정상 렌더됐다.

교훈: **스텁은 실제 응답의 "형태(shape)"를 지켜야 한다.** 안 그러면 검증 대상이 아니라
검증 도구가 깨지고, 그게 진짜 버그처럼 보인다. 동시에 이건 부수 소득이기도 하다 —
API가 예상 밖 형태를 주면 앱 전체가 죽는 취약점을 검증 과정에서 우연히 발견했다.

## 이 기법이 검증하는 것 / 못하는 것

- **검증됨**: UI 분기 로직, 클라이언트 상태 전이, 컴포넌트가 네트워크로 내보내는 요청의
  정확성, 응답을 받은 뒤의 화면 변화
- **검증 안 됨**: 서버가 그 payload를 실제로 받아 저장하는지, S3가 실제로 그 파일을
  받는지. 경계를 가짜로 막았기 때문에 **경계 안쪽(프론트)만** 보장한다.

그래서 이건 실서버 통합 테스트를 대체하지 않는다. "내 프론트 변경이 의도대로 도는가"를
계정·백엔드 없이 혼자 빠르게 확인하는 용도다.
