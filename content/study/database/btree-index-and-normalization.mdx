## 인덱스 — 책의 "찾아보기" 페이지

테이블에 인덱스가 없으면, 원하는 행을 찾기 위해 DB가 테이블 전체를 처음부터 끝까지 훑어야 한다(**Full Scan**). 데이터가 몇 개면 상관없지만, 수십만 행이 되면 매번 느려진다.

**인덱스**는 책의 "찾아보기" 페이지 같은 거다. 본문을 처음부터 다 읽는 대신, 찾아보기에서 "이 단어는 235페이지"라고 바로 알려주는 것처럼, 인덱스는 "이 값은 테이블의 이 위치에 있다"를 미리 정리해둔다.

가장 흔한 구조가 **B-tree**(Balanced Tree)다. 값들을 정렬된 트리 구조로 쌓아둬서, 원하는 값까지 트리를 몇 단계만 타고 내려가면 바로 찾을 수 있다 — 테이블 전체를 훑는 것보다 훨씬 빠르다.

PromptHub의 실제 마이그레이션 파일을 보면:

```sql
CREATE INDEX idx_order_product_order_id
    ON order_product USING btree (order_id);

CREATE INDEX idx_order_product_refunded_at
    ON order_product USING btree (refunded_at);
```

정산 조회 때 `order_id`로, 환불 처리 때 `refunded_at`으로 자주 필터링하니까, 그 컬럼들에 인덱스를 걸어서 조회를 빠르게 만든 것이다.

**주의**: 인덱스는 공짜가 아니다. 데이터를 추가·수정할 때마다 그 인덱스도 같이 갱신해야 해서, 인덱스가 많을수록 쓰기(INSERT/UPDATE)는 느려진다. "자주 조회하는 컬럼에만" 걸어야 하는 이유다.

## 정규화 — 중복을 없애려고 테이블을 나누는 것

**정규화**는 같은 데이터가 여러 곳에 중복 저장되지 않도록 테이블을 쪼개는 설계 원칙이다. 예를 들어 "상품 태그"를 상품 테이블 안에 문자열로 다 몰아넣지 않고, `product_tag`라는 별도 테이블로 분리해서 하나의 태그가 여러 상품과 다대다로 연결되게 하는 식이다.

## PromptHub는 일부러 정규화를 안 한 곳도 있다

PromptHub의 `product` 테이블을 보면:

```sql
CREATE TABLE product (
    -- ...
    tags character varying(255),
    image_urls text,
    -- ...
);
```

`tags`와 `image_urls`는 원래대로라면 `product_tag`, `product_image` 같은 별도 테이블로 정규화할 법한 데이터인데, PromptHub는 그냥 한 컬럼에 몰아 저장한다(`TagsConverter`로 변환).

이건 실수가 아니라 **의도된 비정규화(denormalization)**다. 태그·이미지는 항상 상품 하나와만 묶여서 통째로 읽고 쓰는 데이터라, 별도 테이블로 나누고 JOIN으로 다시 합치는 것보다 한 컬럼에 두는 게 더 단순하고 빠르다. 정규화가 항상 정답은 아니고, "이 데이터를 어떻게 조회·수정하는지"에 따라 오히려 비정규화가 나을 때도 있다.
