## DB 커넥션을 맺는 건 생각보다 비싸다

DB에 연결(커넥션)을 새로 맺는 과정은 TCP 연결, 인증 등 여러 단계를 거쳐서 시간이 꽤 든다. API 요청이 들어올 때마다 매번 새 커넥션을 맺고, 쓰고 나서 바로 끊어버리면 이 비용을 매번 치러야 한다.

**커넥션 풀**은 미리 커넥션을 몇 개 만들어서 "풀"에 담아두고, 요청이 오면 그중 하나를 빌려주고, 다 쓰면 끊지 않고 풀에 반납하는 방식이다.

**비유**: 택시를 매번 새로 사는 대신, 택시 회사가 차 여러 대를 미리 사놓고 손님이 올 때마다 빌려주고 돌려받는 것과 같다. 차를 새로 사는(커넥션을 새로 맺는) 비용을 손님마다 반복하지 않는다.

## HikariCP — Spring Boot의 기본 커넥션 풀

Spring Boot는 별도 설정 없이도 **HikariCP**를 기본 커넥션 풀로 쓴다. HikariCP 자체의 기본값은 이렇다:

- `maximumPoolSize`(풀 안에 최대 몇 개까지 만들지): **10**
- `connectionTimeout`(커넥션을 못 빌리면 얼마나 기다릴지): 30초
- `idleTimeout`(안 쓰는 커넥션을 얼마나 유지할지): 10분

PromptHub는 이 값을 별도로 튜닝하지 않고 **Spring Boot·HikariCP의 기본값을 그대로** 쓴다. 지금 트래픽 규모에서는 기본값으로 충분하다고 판단한 것이다.

## 풀 크기를 왜 함부로 늘리면 안 되나

"동시 요청이 많으니 풀을 크게 늘리면 되지 않나" 싶을 수 있는데, 커넥션 풀 크기는 **DB 서버가 감당할 수 있는 동시 커넥션 수** 안에서 정해야 한다. 앱 서버가 여러 대라면, 각 서버의 풀 크기 × 서버 대수가 DB 쪽 최대 커넥션 수를 넘으면 안 된다.

이전 편(트랜잭션 전파)에서 다룬 `REQUIRES_NEW`도 이 커넥션 풀과 연결된다 — 새로운 독립 트랜잭션을 만들 때마다 별도의 커넥션이 필요해서, 풀 크기가 부족하면 트랜잭션이 커넥션을 못 얻어 대기하다 타임아웃이 날 수 있다.
