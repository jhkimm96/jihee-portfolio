## RDB — 관계와 규칙이 있는 데이터

**RDB**(관계형 DB, PostgreSQL·MySQL 등)는 데이터를 행과 열로 이루어진 테이블에 저장하고, 테이블끼리 외래키(FK)로 관계를 맺는다. "이 컬럼은 반드시 값이 있어야 한다", "이 값은 저 테이블의 값 중 하나여야 한다" 같은 **제약조건**을 DB 차원에서 강제할 수 있다. 지금까지 다룬 트랜잭션·ACID·인덱스·정규화가 전부 RDB의 이야기다.

## NoSQL — 관계·스키마가 느슨한 데이터

**NoSQL**은 "관계형이 아닌" 저장 방식을 통틀어 부르는 말이다. 한 종류가 아니라 여러 갈래가 있다:

- 문서형(MongoDB): 테이블 대신 유연한 구조의 문서(JSON 비슷한 형태)로 저장.
- 키-값형(Redis): 키 하나에 값 하나, 아주 단순하고 빠름.
- 객체 스토리지(S3): 파일 자체를 통째로 저장. 테이블·행 개념이 없다.

공통점은 RDB만큼 엄격한 관계·제약조건을 강제하지 않는 대신, 특정 패턴(대용량 파일, 단순 조회 등)에서 더 단순하고 빠르다는 것.

## PromptHub는 둘 다 쓴다 — 데이터 성격에 맞게 나눠서

PromptHub의 상품 데이터를 보면 이렇게 나뉜다:

- **PostgreSQL(RDB)**: 상품 이름, 가격, 상태, 판매수 같은 정형 데이터. 트랜잭션으로 묶어야 하고, 조회 조건(가격순 정렬, 상태 필터 등)이 다양하다.
- **S3(객체 스토리지)**: PPT·엑셀 같은 상품 원본 파일. 파일 자체는 "이 파일이 저 파일과 관계가 있다"는 개념이 없고, 그냥 통째로 저장했다가 통째로 꺼내 쓰면 된다.

만약 파일까지 PostgreSQL 안에 억지로 넣으려고 하면, 대용량 바이너리 데이터를 관계형 테이블에 우겨넣는 꼴이라 성능도 안 좋고 백업·복제도 무거워진다. PromptHub는 "정형 데이터는 RDB, 파일 원본은 객체 스토리지"로 역할을 나눈 것이다.

## 어떻게 고르나

"이 데이터에 트랜잭션·복잡한 조회·엄격한 관계가 필요한가?"가 기준이다. 필요하면 RDB, 아니면(단순 저장·조회면 충분하면) NoSQL 계열이 더 단순하고 유리하다. 실무에서는 한쪽만 쓰기보다, PromptHub처럼 데이터 성격별로 **섞어 쓰는 경우**가 흔하다.
