## 이름 그대로 — 1번이 N번을 부른다

**전형적인 JPA N+1 패턴**은 이렇다: 상품 목록 10개를 조회하는 쿼리 **1번**을 날린다. 근데 각 상품마다 지연 로딩(LAZY)으로 걸려있는 판매자 정보에 화면에서 접근하는 순간, 그 판매자를 가져오는 쿼리가 상품 개수만큼 **추가로 10번** 나간다. 총 11번의 쿼리 — 이게 **N+1**이다.

```java
List<Product> products = productRepository.findAll();  // 쿼리 1번
for (Product p : products) {
    String sellerName = p.getSeller().getName();  // LAZY라서 접근할 때마다 쿼리 1번씩, 총 N번
}
```

원인은 "지연 로딩 자체"가 아니라, **그 지연 로딩된 값을 반복문 안에서 하나씩 접근하는 패턴**이다.

## PromptHub에서 실제로 발견된 사례

이건 교과서 예시가 아니라, PromptHub의 실제 품질 점검(2026-07-20 스냅샷)에서 나온 발견이다. `ProductQueryService`의 목록 조회 코드를 보면:

```java
Map<UUID, String> sellerNames = products.stream()
    .map(ProductListProjection::sellerId)
    .distinct()
    .collect(Collectors.toMap(id -> id, id -> sellerClient.getSellerInfo(id).sellerName()));
```

이건 JPA 지연 로딩이 아니라 **다른 서비스로의 gRPC 호출**이지만, 패턴은 완전히 똑같다 — 상품 목록을 가져온 뒤, **판매자 수만큼 반복문 안에서 원격 호출을 하나씩** 날린다. DB 쿼리 대신 네트워크 호출이 N번 나가는 것뿐, "1번 조회 후 N번 추가 호출"이라는 근본 구조는 동일하다.

## 어떻게 잡나

**JPA 레벨(고전적인 N+1)**:
- **fetch join**: 처음 조회 쿼리에서부터 연관 데이터를 JOIN으로 한 번에 같이 가져온다.
- **`@EntityGraph`**: 어떤 연관관계를 즉시 같이 가져올지 쿼리 메서드에 선언적으로 지정한다.
- **batch size 설정**: 반복문 안에서 하나씩 가져오는 대신, "한 번에 N개씩 묶어서" 가져오도록 설정한다.

**PromptHub의 gRPC 사례 같은 원격 호출 레벨**:
- **배치 조회 API로 교체**: "판매자 하나씩 조회"가 아니라 "판매자 ID 목록을 한 번에 보내서 한 번의 호출로 다 받아오는" API로 바꾼다. PromptHub도 이 방식(배치 조회)으로 교체하는 게 품질 점검의 권장 사항으로 나왔다.

핵심은 같다 — **"필요한 걸 하나씩 여러 번" 대신 "필요한 걸 한 번에 묶어서"** 가져오는 것.
