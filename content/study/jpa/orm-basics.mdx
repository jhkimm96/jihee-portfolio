## SQL을 직접 안 쓰고 뭘 대신 쓰나

자바 코드는 객체(클래스와 필드)로 데이터를 다루고, DB는 테이블(행과 열)로 데이터를 저장한다. 원래는 자바 객체를 저장하려면 `INSERT INTO product (...) VALUES (...)` 같은 SQL을 매번 손으로 써야 한다. 객체 필드가 하나 늘어나면 그 SQL도 다 고쳐야 한다.

**ORM**(Object-Relational Mapping)은 이 반복 작업을 대신해주는 기법이고, **JPA**(Java Persistence API)는 자바에서 이 ORM을 쓰기 위한 표준 명세다. **Hibernate**는 그 명세를 실제로 구현한 라이브러리 중 하나고(Spring Boot가 기본으로 쓰는 게 Hibernate다).

**비유**: 통역사가 있으면 한국어로 말하고 상대는 영어로 듣는다. JPA는 "자바 객체 언어"와 "SQL 언어" 사이의 통역사다 — 개발자는 자바 객체만 다루면, JPA가 알아서 그에 맞는 SQL을 만들어 실행한다.

## PromptHub의 실제 예시

PromptHub의 `Product` 클래스는 이렇게 시작한다:

```java
@Entity
@Table(name = "product", schema = "product_service")
public class Product {

    @Id
    @Column(name = "id", nullable = false)
    private UUID id;

    @Column(name = "name", nullable = false, length = 200)
    private String name;

    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false)
    private ProductStatus status;
    // ...
}
```

- `@Entity` — "이 클래스는 DB 테이블과 매핑되는 자바 객체다"라는 표시.
- `@Table(name = "product", schema = "product_service")` — 정확히 어느 스키마의 어느 테이블과 연결되는지 지정.
- `@Id` — 이 필드가 그 테이블의 기본키(PK)다.
- `@Column` — 자바 필드 이름과 실제 DB 컬럼 이름·제약조건을 연결.

이 클래스 하나만 만들어두면, `productRepository.save(product)`라고 자바 코드로만 호출해도 JPA가 알아서 `INSERT`나 `UPDATE` SQL을 만들어 실행한다. 개발자가 SQL 문자열을 직접 조립할 필요가 없다.

## 이게 왜 편한가 — 그리고 뭘 조심해야 하나

컬럼이 하나 추가되면 `@Column` 하나 추가하면 끝이다. SQL을 여기저기서 고칠 필요가 없다. 대신 JPA가 "알아서" 해주는 부분이 많아질수록, 실제로 어떤 SQL이 나가는지 개발자가 모르고 넘어가기 쉽다. 나중 편(영속성 컨텍스트, N+1 문제)에서 다루는 문제들이 대부분 "JPA가 알아서 해주는 부분을 제대로 이해 못 해서" 생긴다.
