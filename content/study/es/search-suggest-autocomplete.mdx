지금까지는 "검색 버튼을 누른 뒤"의 이야기였다. 이 노트는 그 **앞단** — 사용자가 검색창에 **치는 중** 도와주는 두 가지다.

- **자동완성** — "무선"까지 쳤을 때 "무선 이어폰", "무선 마우스"를 추천.
- **오타 교정** — "아이푠"이라 쳤을 때 "아이폰 찾으세요?"로 바로잡기.

## 자동완성 — 세 가지 접근법

- **Completion Suggester** — ES가 자동완성 전용으로 제공하는 기능. `completion` 필드에 저장하고, 메모리(FST) 기반이라 **매우 빠르다.** 접두어(prefix) 매칭이 핵심.
- **edge n-gram** — 색인할 때 단어의 앞 조각들을 미리 만들어 둔다("이어폰" → `이`,`이어`,`이어폰`). 그럼 일반 `match` 검색으로도 접두어 자동완성이 된다. 유연하지만 색인이 커진다.
- **search_as_you_type 필드** — edge n-gram을 자동으로 만들어주는 편의 필드 타입. "치는 대로 매칭"에 맞춰져 있다.

가장 기본은 Completion Suggester다.

## Completion Suggester 사용법

`completion` 타입 필드를 따로 두고, 자동완성으로 뜰 문구를 `input`에 넣는다.

```json
// 매핑: 자동완성 전용 필드
PUT products
{
  "mappings": {
    "properties": {
      "name":         { "type": "text", "analyzer": "korean" },
      "name_suggest": { "type": "completion" }
    }
  }
}

// 색인: input에 뜰 문구들, weight로 우선순위
POST products/_doc
{ "name": "무선 이어폰",
  "name_suggest": { "input": ["무선 이어폰", "이어폰"], "weight": 10 } }
```

```json
// 자동완성 요청: prefix로 검색
POST products/_search
{
  "suggest": {
    "product-suggest": {
      "prefix": "무선",
      "completion": { "field": "name_suggest" }
    }
  }
}
```

- **`input`** — 이 문서를 어떤 문구로 자동완성에 띄울지. 여러 개 가능("무선 이어폰"과 "이어폰" 둘 다로 뜨게).
- **`weight`** — 우선순위. **인기 검색어·판매량을 weight로** 주면 인기순 자동완성이 된다.
- prefix 매칭이라 "무선"으로 치면 뜨지만, 중간 단어("이어폰")로 띄우려면 그 문구도 `input`에 넣어야 한다.

**자동완성과 오타교정이 다르다.**

- **자동완성(Completion Suggester)** — 그렇다. 사용자가 칠 때마다 요청이 나간다("무"→"무선"). 그래서 Completion Suggester가 인메모리(FST)로 아주 빠르게 설계됐다 — 거의 매 키 입력마다 불리니까.
- **오타교정(Term/Phrase Suggester)** — 다르다. 타이핑 중이 아니라 **검색을 실행한 뒤** 결과가 적거나 0건일 때 한 번 불러 "이거 찾으셨나요?"를 띄운다.

매 키마다 그대로 쏘면 요청이 폭주하므로 실무에선 **debounce**(입력이 멈추면 ~200ms 후 1회 호출) + **최소 글자수**(2자 이상부터) + 오래된 요청 취소를 건다. 흐름은 일반 검색과 같은 계층이다:

```
타이핑 → (debounce) → GET /api/products/suggest?q=무선
       → 백엔드가 ES suggest API 호출 → 결과를 드롭다운으로
```

## 한글 자동완성의 진짜 문제 — 초성·자모

여기가 한글의 난점이다. Completion Suggester는 **완성된 글자의 접두어**만 매칭한다. 그래서:

- "삼성" 자동완성: "삼"까지 치면 된다. 하지만…
- **초성 "ㅅㅅ"** 으로 "삼성"을 띄우거나, **조합 중간 "삼ㅅ"** 상태로 매칭하는 건 **기본 기능으로 안 된다.**

이유: 한글은 자모(ㅅ+ㅏ+ㅁ…)가 조합된 글자라, "삼"과 "ㅅ"은 ES 입장에서 완전히 다른 문자다. 그래서 한글 초성 자동완성을 하려면 **한글을 자모로 분해해서 색인**해야 한다.

- **개념**: 색인 시 "삼성" → 초성 `ㅅㅅ`, 자모 `ㅅㅏㅁㅅㅓㅇ`으로 분해한 값을 함께 저장 → 여기에 edge n-gram을 걸어 접두어 매칭. 그럼 "ㅅㅅ", "삼ㅅ", "삼서"가 다 걸린다.
- **구현**: 이 자모 분해는 ES 기본에 없어서, **커스텀 문자 필터(char filter)나 커뮤니티 플러그인**으로 직접 구성한다. (3번 노트의 nori는 형태소 분석용이지 자모 분해는 하지 않는다.)

정직하게: **한글 초성 자동완성은 표준 기능만으로는 안 되고, 자모 분해 전처리를 직접 설계해야 하는 대표적 난이도 구간**이다. 상품 검색 자동완성 품질의 핵심이자 공수가 드는 부분.

## 오타 교정 — Term / Phrase Suggester

잘못 친 검색어를 "이거 찾으셨나요?"로 바로잡는다.

- **Term Suggester** — **단어 단위** 교정. **편집 거리(edit distance)** 로 "철자가 가까운 단어"를 제안한다. "아이푠" → "아이폰".

```json
POST products/_search
{
  "suggest": {
    "typo": {
      "text": "아이푠",
      "term": { "field": "name" }
    }
  }
}
```

- **Phrase Suggester** — **문장/문맥** 단위. 여러 단어의 그럴듯한 조합을 본다. "무선 이어펀" 전체를 문맥으로 "무선 이어폰"으로.

```json
POST products/_search
{
  "suggest": {
    "phrase-check": {
      "text": "무선 이어펀",
      "phrase": { "field": "name.trigram", "size": 1 }
    }
  }
}
```

**한영/영한 오타**(키보드 배열: "무선"을 영문 상태로 "antjs"처럼 침)는 편집 거리로는 못 잡는다. 이건 Suggester가 아니라 **키보드 배열 변환 로직**(문자 필터/전처리)으로 따로 처리한다.

## 상품 검색에 어떻게 쓰나 (종합)

- **자동완성** = `completion` 필드(기본) 또는 **자모 분해 + edge n-gram**(한글 초성까지 원하면). `weight`에 인기·판매량 반영.
- **오타 교정** = 검색 결과가 적거나 0건일 때 **Term Suggester**로 "이거 찾으셨나요?" 노출.
- **한영 오타** = 별도 키보드 변환 전처리.
- 자동완성용 데이터는 보통 **상품명 + 인기 검색어 로그**로 만든다 — 그 "인기 검색어"가 어디서 나오냐면, 다음 노트의 **집계(6번)**와 **검색 로그(7번)**다.

즉 좋은 자동완성은 "행동 로그 → 집계 → 자동완성 weight"로 순환한다. 그래서 다음은 **집계(6번 노트)**다.
