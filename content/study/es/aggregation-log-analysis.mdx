앞의 노트들은 "문서를 찾는" 이야기였다. 이 노트는 "찾은 걸 **세고·묶고·통계 내는**" 집계다. **인기 검색어, 검색량 추이, 순 방문자수** — 당신이 말한 "로그 분석"이 전부 여기서 나온다. Kibana 대시보드의 막대·선 그래프도 속으로는 이 집계를 돌린 결과다.

## 집계란

**집계(aggregation)**는 검색 결과 문서들 위에서 **통계를 계산**하는 것이다. 검색(어떤 문서?)과 별개로, `aggs`로 요청한다.

```json
GET search-logs/_search
{
  "size": 0,
  "aggs": {
    "popular_queries": {
      "terms": { "field": "query.keyword", "size": 10 }
    }
  }
}
```

`"size": 0`은 **문서 본문은 받지 않고 집계 결과만** 달라는 뜻(로그 분석은 개별 문서가 아니라 통계가 필요하니까).

<details>
<summary><b>🔍 size:0은 무슨 뜻이야?</b></summary>

`_search`는 기본으로 매칭된 문서 10건을 함께 돌려준다. 하지만 로그 분석은 개별 문서가 아니라 **집계 결과(통계)**만 필요하다. `"size": 0`은 "문서 본문은 주지 말고 집계만 계산해"라는 뜻이다. 수백만 건 로그에서 문서까지 실어 보내면 느리고 낭비라, 집계 요청엔 거의 항상 붙인다.

</details>

## 세 종류

- **metric(메트릭)** — 숫자 하나로 요약. 합계·평균·고유수 등.
- **bucket(버킷)** — 조건으로 **그룹을 나눔.** "카테고리별", "날짜별". 각 그룹이 버킷.
- **pipeline(파이프라인)** — 다른 집계의 **결과 위에 다시** 집계. 이동평균, 버킷 정렬 등.

## Metric 집계 — 숫자 통계

- `sum`·`avg`·`min`·`max` — 합계·평균·최소·최대. (매출 합, 평균 가격)
- `stats` — 위 넷을 한 번에.
- **`cardinality`** — **고유값 개수.** `userId`에 걸면 **순 방문자수(중복 제거)**.
- `percentiles` — 백분위. "응답시간 95%가 몇 ms 이하인가".

```json
// 순 방문자수
{ "size": 0, "aggs": { "unique_users": { "cardinality": { "field": "userId" } } } }
```

## Bucket 집계 — 그룹으로 묶기

여기가 로그 분석의 핵심이다.

- **`terms`** — 값별로 묶고 개수 세기. **인기 검색어 TOP 10**, 카테고리별 상품 수.
- **`date_histogram`** — 시간 구간별로 묶기. **검색량 시간대별 추이**(트렌드).
- `range`·`date_range` — 직접 정한 구간(가격대 0~1만/1~5만, 기간별).
- `histogram` — 일정 간격 숫자 구간(가격 1만원 단위).

```json
// 인기 검색어 TOP 10
{ "size": 0,
  "aggs": { "popular": { "terms": { "field": "query.keyword", "size": 10 } } } }

// 검색량 일별 추이
{ "size": 0,
  "aggs": { "daily": { "date_histogram": { "field": "ts", "calendar_interval": "day" } } } }
```

주의: `terms`·정렬은 `keyword`나 숫자 필드에 쓴다(`text`는 분석돼서 안 됨 → `query.keyword`, 2번 노트). 시간 구간은 `calendar_interval`("day"/"week"/"month") 또는 `fixed_interval`("30m"/"1h")로 준다.

<details>
<summary><b>🔍 calendar_interval이랑 fixed_interval 차이는?</b></summary>

`date_histogram`이 시간을 어떤 단위로 자를지 정한다.

- **`calendar_interval`** — 달력 단위. `"day"`,`"week"`,`"month"`,`"year"`. 달마다 날짜 수가 달라도(28~31일) "한 달"로 정확히 자른다.
- **`fixed_interval`** — 고정 길이. `"30m"`,`"1h"`,`"7d"`. 항상 정확히 그 길이. "30분 단위"처럼 달력과 무관한 구간에.

일·주·월 트렌드는 `calendar_interval`, 분·시간 단위 세밀 분석은 `fixed_interval`을 쓴다. (예전엔 `interval` 하나였는데 헷갈림 때문에 지금은 이 둘로 나뉘어 있다.)

</details>

## 중첩 집계 — 버킷 안에 통계

버킷 안에 또 집계를 넣어 "그룹별 통계"를 낸다. 로그 분석에서 제일 많이 쓰는 형태다.

```json
// 카테고리별 평균 가격
{ "size": 0,
  "aggs": {
    "by_category": {
      "terms": { "field": "category" },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } }
      }
    }
  }
}
```

"카테고리로 묶고(bucket), 각 그룹의 평균 가격(metric)". "일별로 묶고, 각 날의 순 방문자수" 같은 조합이 대시보드의 대부분이다.

<details>
<summary><b>🔍 중첩 집계, 어떻게 읽어?</b></summary>

버킷 집계 안에 다시 `aggs`를 넣으면 "그룹을 나눈 뒤, 각 그룹 안에서 또 집계"가 된다. 바깥에서 안으로 읽는다.

```json
"by_category": {
  "terms": { "field": "category" },              // ① 카테고리로 그룹을 나눔
  "aggs": { "avg_price": { "avg": { "field": "price" } } }  // ② 각 그룹 안에서 평균 가격
}
```

읽으면 "① 카테고리별로 묶고 → ② 각 묶음의 평균 가격". 결과도 카테고리 버킷마다 `avg_price`가 붙어 나온다. "일별로 묶고 → 각 날의 순방문자수"처럼 계속 중첩할 수 있다.

</details>

## Pipeline 집계 — 집계 결과 위의 집계

이미 나온 버킷 결과를 다시 가공한다. 예: 일별 매출(date_histogram) → **누적 합계**(`cumulative_sum`), **이동 평균**(`moving_fn`), **상위 버킷 정렬**(`bucket_sort`). 트렌드 그래프에 추세선을 얹을 때 쓴다. (자주 쓰진 않으니 필요할 때 찾아 쓰면 된다.)

## 근삿값 주의 — 분산 환경의 함정

**`cardinality`와 `terms`는 분산 환경에서 근삿값**이다. 샤드마다 부분 집계를 낸 뒤 합치는데, 이때 오차가 생긴다.

- `cardinality` — HyperLogLog 알고리즘의 근사. 대부분 오차가 미미하고 `precision_threshold`로 조절.
- `terms` — 샤드마다 상위 N개만 모아 합치므로, 하위권 순위·개수가 살짝 부정확할 수 있다. `shard_size`를 키우면 정확도 ↑(비용 ↑).

즉 "인기 검색어 TOP 10"은 믿을 만하지만, "정확히 몇 번"이나 하위권 순위는 근삿값임을 감안한다.

<details>
<summary><b>🔍 cardinality가 왜 근삿값이야? HyperLogLog·precision_threshold가 뭔데?</b></summary>

"고유 userId가 몇 개인가"를 **정확히** 세려면 모든 userId를 메모리에 담아 중복을 제거해야 한다. 사용자가 수천만이면 메모리가 터진다. 그래서 ES는 **HyperLogLog** 알고리즘을 쓴다 — 값들을 통계적으로 요약해서 **적은 메모리로 "대략 몇 개"를 추정**한다(보통 오차 1% 안팎).

`precision_threshold`는 "이 개수 이하까지는 정확히, 그 위는 근사"의 경계다. **값을 키우면**(예: 40000) 더 정확해지지만 **메모리를 더 쓴다.** 순 방문자수처럼 오차가 조금 있어도 되는 지표는 기본값으로 충분하다.

</details>

<details>
<summary><b>🔍 terms는 왜 부정확할 수 있어? shard_size를 키운다는 게 뭐야?</b></summary>

데이터는 여러 샤드(1번 노트)에 흩어져 있다. "인기 검색어 TOP 10"을 낼 때 ES는 **각 샤드에서 상위 몇 개씩만 모아** 합친다. 어떤 검색어가 한 샤드에선 11위라 안 뽑혔는데 다 합치면 사실 8위일 수 있다 → 하위권 순위가 살짝 틀릴 수 있다.

`shard_size`는 "각 샤드에서 몇 개를 모을지"다. **값을 키우면**(TOP 10을 원해도 샤드마다 100개씩 모으면) 합칠 재료가 많아져 **더 정확**해진다. 대신 **비용(메모리·속도)이 는다.** 그래서 TOP 10은 대체로 믿을 만하고, 정밀 순위가 필요할 때만 키운다.

</details>

## 행동 로그 분석에 어떻게 쓰나 (종합)

검색·클릭·구매 이벤트를 색인해두면(7번 노트), 이 집계들로 분석이 나온다.

| 분석 | 집계 |
|---|---|
| 인기 검색어 TOP N | `terms` (query.keyword) |
| 검색량 시간대별 추이 | `date_histogram` (ts) |
| 순 방문자수 | `cardinality` (userId) |
| 결과 0건 검색어(개선 대상) | `terms` + resultCount=0 필터 |
| 카테고리별 판매·조회 | `terms` + 하위 `sum`/`avg` |

**Kibana 대시보드의 그래프 하나하나가 이 집계 요청**이다. 그래서 앞서 말한 "ELK 대시보드에 로그 보내주기"는 = 이벤트를 잘 색인해두면, 대시보드 담당자가 이 집계들로 그림을 그린다는 뜻이다.

<details>
<summary><b>🔍 그 로그는 실제로 어디에 어떻게 저장돼? ES에 HTTP로 보내나?</b></summary>

**ES엔 "테이블"이 아니라 인덱스가 있다.** 로그 이벤트 한 건 = **문서 한 건**, 그 문서들이 모인 게 인덱스다(2번 노트 대응표). 로그는 보통 **날짜별 인덱스**(`search-logs-2026.07.15`)에 쌓인다.

넣는 법: ES는 9200 포트에 **HTTP REST API**가 있어서, 이벤트 JSON을 **HTTP POST**로 보낸다.

```json
POST http://es:9200/search-logs-2026.07.15/_doc
{ "eventType":"search", "query":"무선 이어폰", "userId":"u123", "resultCount":42, "ts":"..." }
```

서비스가 직접 POST 하거나, 로그파일/Kafka → Logstash·Filebeat가 대신 넣는다.

**ES ↔ 프로젝트는 1:1이 아니다.** 한 ES 클러스터에 여러 인덱스가 산다 — `products`(상품 검색), `search-logs-*`(검색 로그) 등. 서비스마다 ES를 따로 두는 게 아니라 한 클러스터를 공유하고 데이터 종류별로 인덱스를 나눈다. (직접 vs Kafka 경유, 날짜별 인덱스·ILM 같은 파이프라인 상세는 7번 노트.)

</details>

<details>
<summary><b>🔍 ELK가 뭐고 Kibana가 뭐야? 무슨 흐름으로 대시보드가 만들어져?</b></summary>

**ELK는 세 도구를 묶은 스택 이름**이다: **E**lasticsearch + **L**ogstash + **K**ibana.

- **Elasticsearch** — 로그를 저장·검색·집계 (지금 배우는 것)
- **Logstash**(또는 Beats) — 로그를 모아 ES로 실어 나르는 운반책
- **Kibana** — ES에 쌓인 걸 **그래프·대시보드로 보여주는 화면**

그래서 "**ELK 대시보드**"와 "**Kibana 대시보드**"는 사실상 같은 걸 가리킨다 — 대시보드를 그리는 도구가 Kibana이고, ELK는 그 셋의 묶음 이름일 뿐이다.

```mermaid
flowchart LR
    App["① 서비스<br/>이벤트 발생"] --> Ship["② Logstash/Beats<br/>수집·운반"]
    Ship --> ES["③ Elasticsearch<br/>저장·집계"]
    Kibana["④ Kibana<br/>대시보드"] -. 집계 요청 .-> ES
    ES -. 집계 결과 .-> Kibana
```

무엇이 만들어지나: 서비스가 남긴 이벤트가 ES에 쌓이고, Kibana에서 막대·선 그래프를 만들면 Kibana가 그때그때 ES에 **이 노트의 집계(terms·date_histogram 등)**를 요청해 그림을 그린다. 즉 대시보드 그래프 하나 = 집계 요청 하나다.

</details>

그럼 그 이벤트를 **어떻게 ES로 흘려보내나** — 색인 파이프라인·시계열 인덱스가 **7번 노트**다.
