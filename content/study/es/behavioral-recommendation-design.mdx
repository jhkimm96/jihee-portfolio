7번 노트에서 행동 이벤트를 ES에 쌓았다. 이제 그 위에 **추천**을 얹는다. 핵심 통찰 하나로 시작한다: **추천은 여러 종류지만, 전부 같은 "행동 이벤트 데이터"에서 나온다.** 각각을 새로 짜는 게 아니라, 이벤트를 잘 남겨두면 그 위의 변주다.

## 추천은 한 종류가 아니다

| 추천 종류 | 무슨 데이터로 |
|---|---|
| 개인 추천 | 그 유저의 조회·구매 이력 |
| 함께 본 상품 | 같은 세션에서 연달아 본 상품 쌍의 빈도 |
| 비슷한 상품 | 태그·카테고리(lexical) 또는 벡터 유사도 |
| 비슷한 셀러/유저 | 취향 벡터의 유사도(9번 노트) |
| 인기순 | 전체 조회·구매·검색 집계 |

**바닥(이벤트 데이터)이 튼튼한 게 핵심**이고, 추천 종류는 그 위의 변주다.

## 역할 분리 — 이게 제일 중요

추천에서 헷갈리기 쉬운 지점: **추천 "계산"과 ES의 "검색·랭킹"은 다른 층이다.**

- **추천 알고리즘 자체**(협업필터링 계산, 벡터 임베딩 학습) — 무거운 ML이라 **ES 밖**의 별도 추천 서비스/모델이 담당.
- **ES가 하는 일** — 그 알고리즘이 만든 **결과(추천 후보·벡터)를 저장**하고, 요청이 오면 **빠르게 찾아 순위를 매기는 것.**

비유: **모델 = 취향 좌표를 계산하는 계산기**, **ES = 그 좌표로 가까운 걸 순식간에 찾는 검색기.** 이 분리 덕에 모델을 바꿔도 ES는 그대로 쓴다. (당신이 ML 모델을 직접 만들 필요는 없다 — ELK 대시보드처럼 별도 담당.)

<details>
<summary><b>🔍 역할 분리 — 추천모듈이 알고리즘을 쿼리에 실어? kNN은 누가?</b></summary>

알고리즘을 쿼리에 싣는 게 아니라, **알고리즘의 결과물(벡터)**을 싣는다. 나눠 보면:

- **추천모듈** — 유저·상품을 **벡터(취향 좌표)로 "만든다"**. 무거운 ML이라 미리 계산해 둠. 상품 벡터는 ES products 인덱스에 **저장**.
- **검색 시** — 유저 벡터 **하나만 쿼리에 실어** ES로 보낸다.
- **ES** — 그 벡터로 **kNN(가까운 이웃 찾기)을 실행**해 유사 상품을 돌려준다.

"벡터를 만드는 것 = 추천모듈, 벡터로 가까운 걸 찾는 것(kNN) = ES." kNN은 벡터 유사도 검색이 맞고(9번 노트), 그 실행 주체가 ES다. 모델을 바꿔도 ES kNN은 그대로 쓴다 — 이래서 둘을 나눈다.

</details>

<details>
<summary><b>🔍 nori+태그로 "비슷한 상품"도 되잖아? 왜 벡터가 필요해?</b></summary>

맞다. 상품엔 태그·카테고리·설명이 있으니 **"비슷한 상품"도 lexical하게 할 수 있다** — ES엔 `more_like_this`("이 문서와 비슷한 문서 찾기") 쿼리가 있고, 공유 태그·카테고리로도 된다. 태그가 잘 붙어 있으면 싸고 빠르고 충분하다. (벡터가 "더 가벼워서" 쓰는 게 아니다 — 오히려 임베딩 생성·저장 인프라가 더 무겁다.)

벡터를 쓰는 진짜 이유는 **속도가 아니라, 태그·단어가 표현 못 하는 유사성**이다:

- **태그 너머의 의미·느낌 유사** — 태그는 사람이 붙인 거라 거칠다. "스타일이 비슷", "용도가 비슷"을 태그가 다 못 담는다. 임베딩은 이름·설명(때론 이미지)에서 잡는다.
- **비교할 대상이 아예 없는 경우** — "이 유저 취향에 맞는 상품"은 매치할 상품 문서가 없다. 유저 취향을 벡터로 만들어야 찾는다. 여기가 벡터가 꼭 필요한 지점.

정리: 태그가 잘 되어 있고 충분하면 `more_like_this`(lexical)가 맞다. 벡터는 "태그가 못 잡는 유사성"이나 "유저 취향 매칭"이 필요할 때 값어치가 난다. 하이브리드는 이 둘을 합치는 것.

</details>

## MSA에서 추천 모듈은 데이터를 어떻게 얻나

MSA는 각 서비스가 자기 DB만 소유한다. 추천 모듈은 **남의 테이블을 직접 안 읽는다.** 대신:

- **행동 데이터** → Kafka 이벤트로 받는다(7번 노트).
- **상품 정보** → product-service API로 물어보거나, ES products 인덱스(검색용 사본)를 본다.

```
product-service ─이벤트→ Kafka ─→ recommendation 모듈
                                     └ 추천 계산 → 결과를 ES에 저장·조회
상품 원본 ─────────────→ ES products 인덱스(사본)
```

## 개인화 — function_score로 랭킹 조정

가장 실용적인 시작점은 4번 노트의 `function_score`다. 검색·목록 결과를 **그 유저의 행동 신호로** 재정렬한다.

- 최근 본 카테고리를 부스팅 (`filter` + `weight`)
- 산 적 있는 셀러/브랜드 가점
- 인기(판매량·조회수)를 `field_value_factor`로 반영

```json
GET products/_search
{
  "query": {
    "function_score": {
      "query": { "match_all": {} },
      "functions": [
        { "filter": { "term": { "category": "전자기기" } }, "weight": 2 },   // 최근 본 카테고리
        { "field_value_factor": { "field": "salesCount", "modifier": "log1p" } }
      ],
      "boost_mode": "multiply"
    }
  }
}
```

여기서 "전자기기", "salesCount 중요도" 같은 **개인화 신호는 추천 모듈/유저 프로필에서** 와서 쿼리에 실린다(규칙은 저장 안 되고 매 요청에 실림, 4번 노트).

<details>
<summary><b>🔍 개인화 신호가 "쿼리에 실린다"는 게 무슨 말? 벡터로만 가능해?</b></summary>

**개인화는 벡터로만 하는 게 아니다.** 두 가지 별개 경로가 있다:

- **규칙 기반 (function_score)** — 벡터 없이. 유저 프로필을 읽어 `filter`+`weight`("최근 본 전자기기 가점"), `field_value_factor`("인기 반영")를 쿼리에 채운다. 개인화의 시작점.
- **벡터 기반 (kNN)** — 유저 취향 벡터를 별도 `knn` 쿼리에 채운다. 의미 기반 고도화(9번 노트).

둘 다 **"쿼리에 실린다" = 저장된 규칙이 아니라, 요청을 만들 때 그 유저 값을 채워 보낸다**는 뜻이다. 왜? **유저마다 신호가 다르니까.** 같은 목록 화면이라도 유저 A는 "전자기기 가점", 유저 B는 "패션 가점"이 실려 서로 다른 쿼리가 나간다. (function_score엔 `filter`·`weight`를, 벡터는 별도 `knn` 쿼리에 채운다 — 둘은 다른 경로다.)

</details>

<details>
<summary><b>🔍 판매량·검색량(인기)을 추천에 어떻게 섞어?</b></summary>

인기는 **독립 추천이자, 다른 추천에 얹는 신호**다. function_score의 `field_value_factor`로 넣는다.

```json
"functions": [
  { "field_value_factor": { "field": "salesCount",  "modifier": "log1p" } },
  { "field_value_factor": { "field": "searchCount", "modifier": "log1p" } }
],
"score_mode": "sum", "boost_mode": "multiply"
```

`log1p`로 완만하게(1등이 100배 먹지 않게). 개인화·유사도 결과에 이걸 곱하면 "비슷하면서 인기 있는" 추천이 된다. 콜드 스타트(데이터 없는 신규 유저)엔 인기만으로 폴백.

**그 숫자는 어디서?** 행동 로그(7번) → 집계(6번)로 상품·검색어별 인기를 계산 → 그 값을 상품 문서의 `salesCount`·`searchCount` 필드로 갱신(이벤트로 +1 하거나 배치로 집계값 반영). 그래서 "로그 → 집계 → 인기 점수 → function_score 부스팅"으로 한 바퀴 돈다.

</details>

## 추천 종류별 구현 개요

- **개인 추천** — 유저 프로필(최근 본 카테고리·구매 브랜드)을 function_score 신호로. 콜드 스타트(신규 유저)엔 인기순 폴백.
- **함께 본 상품** — 같은 `sessionId`에서 **동시 출현한 상품 쌍**을 집계(배치로 미리 계산)해 "이 상품 → 함께 본 상품 목록"을 만들어 둔다. 조회 시 그 목록을 상품으로 붙인다.
- **비슷한 상품** — 태그·카테고리 기반 `more_like_this`(lexical) 또는 벡터 유사도(9번 노트).
- **비슷한 셀러/유저** — 취향을 **벡터**로 만들어 유사도로 찾는다. ES의 kNN이 여기 쓰인다(9번 노트).
- **인기순** — 6번 노트의 집계(terms + 기간 필터). 콜드 스타트·기본 정렬의 폴백.

<details>
<summary><b>🔍 "함께 본 상품"은 어떻게 만들어?</b></summary>

같은 `sessionId`(또는 유저)가 **연달아 본 상품 쌍**을 센다. A와 B가 같은 세션에 자주 함께 등장하면 "A → B" 관련이 높다. 이걸 **배치로 미리 계산**(예: 밤마다 최근 로그 집계)해서 "상품 → 함께 본 상품 목록"을 만들어 둔다.

실시간으로 매번 세는 게 아니라 **미리 만들어 저장**해두고, 상품 상세를 열 때 그 목록의 productId들을 상품 정보로 붙여 보여준다. 장바구니·구매 동시출현으로 확장하면 "이거 산 사람이 함께 산 상품"이 된다.

</details>

<details>
<summary><b>🔍 콜드 스타트가 뭐야?</b></summary>

신규 유저·신상품은 **행동 데이터가 없어서** 개인화·벡터 추천을 할 수 없다. 이걸 콜드 스타트 문제라 한다. 해법은 **폴백(대체)**: 데이터가 쌓이기 전엔 개인화 대신 **인기순·신상품·카테고리 베스트**를 보여준다(6번 집계로 계산). 행동이 쌓이면 점차 개인화로 전환한다. 그래서 인기순 추천은 "기본기"이자 콜드 스타트 안전망이다.

</details>

## 추천은 productId로, 이름은 나중에

흔한 오해: 추천이 상품명으로 도는 게 아니다.

- 추천 **계산**은 이름을 안 본다. **productId 단위**로 "비슷한 벡터"나 "함께 본 빈도"로 후보를 고른다.
- 결과는 **productId 목록**: `["p456", "p789", ...]`
- **name·가격은 마지막에** 그 productId로 조회해 붙인다(화면 표시용).

즉 **추천 로직 = 숫자·행동으로 productId 뽑기**, **name = 결과 꾸미기용 조회.** 둘은 분리된다.

<details>
<summary><b>🔍 왜 이름 말고 productId로 추천해?</b></summary>

이름은 바뀔 수 있고 중복될 수 있지만 **productId는 고유·불변**이라 정확히 한 상품을 가리킨다. 벡터·동시출현 계산도 전부 이 id 단위로 돈다. 그래서 추천 파이프라인 내부는 id로만 움직이고, 사람이 보는 이름·가격·이미지는 **맨 마지막에** id로 조회해 붙인다. 이렇게 하면 상품 정보가 바뀌어도 추천 로직은 안 건드린다.

</details>

## 이 시스템에 어떻게 쓰나 (종합)

```
행동 이벤트(7번) ─→ 집계·모델(6번 + 추천모듈)
                        ├ 개인화 신호·인기 → function_score(4번)로 검색·목록 재정렬
                        ├ 함께 본 상품 → 세션 동시출현 배치 집계
                        └ 취향 벡터 → ES kNN(9번)으로 유사 상품·셀러
결과 productId 목록 ─→ 상품 정보 붙여 화면에
```

- **당신** = 이벤트 잘 남기기(7번) + function_score로 랭킹 조정 + 추천 결과를 ES에서 조회해 서빙.
- **남이** = 무거운 ML 모델·벡터 임베딩.

function_score까지가 "세어서 하는 추천"이라면, 다음은 "의미로 하는 추천" — **벡터 검색(kNN), 9번 노트**다.
