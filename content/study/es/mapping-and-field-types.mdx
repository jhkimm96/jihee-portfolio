1번 노트에서 "원본은 RDB, 검색은 ES"라고 나눴다. (RDB = MySQL·PostgreSQL 같은, 흔히 쓰는 그 테이블 DB)
이제 ES 쪽 상품 인덱스를 어떻게 생기게 할지 — **매핑(mapping)**을 정한다. 매핑을 잘못 잡으면 검색이 안 되거나 나중에 통째로 재색인해야 하므로, 여기가 설계의 뼈대다.

## 매핑이란

매핑은 **인덱스의 각 필드가 어떤 타입인지, 어떻게 색인될지를 정의**한 것이다. RDB의 테이블 스키마와 비슷하지만 두 가지가 다르다.

- **필드 타입에 따라 "색인 방식"이 달라진다.** 같은 문자열이라도 `text`로 잡으면 분석기로 쪼개 전문 검색용이 되고, `keyword`로 잡으면 통째로 저장돼 정확 매칭·집계용이 된다. (이게 이 노트의 핵심)
- **한 번 정한 필드 타입은 못 바꾼다.** 뒤에서 다시 다룬다.

<details>
<summary><b>🔍 그럼 RDB를 써야 하나? ES 매핑이랑 RDB 테이블 만드는 거랑 뭐가 다르지?</b></summary>

**둘 중 하나를 고르는 게 아니라 둘 다 쓴다.** 1번 노트의 "원본은 RDB, 검색은 ES" 그대로다.

- **RDB 테이블** = 상품의 **원본**(가격·재고·판매자·주문). 정합성·트랜잭션·조인이 필요한 진짜 데이터. 이건 그대로 RDB에 둔다.
- **ES 상품 인덱스** = 그 원본을 **검색·추천용으로 복제**한 사본. RDB를 대체하는 게 아니라 옆에 두는 "검색 전용 창구"다.

겉보기엔 둘 다 "필드 정의"라 비슷하지만 목적이 다르다:

| | RDB 테이블 | ES 매핑(인덱스) |
|---|---|---|
| 목적 | 원본 저장·거래(트랜잭션) | 빠른 검색·집계 |
| 정의하는 것 | 값을 어떻게 **저장·제약**할지 | 필드를 어떻게 **검색 가능하게 쪼갤지** |
| 구조 | 정규화(여러 테이블 + 조인) | 비정규화(한 문서에 다 담기) |
| 트랜잭션 | 있음 | 없음 |

그래서 상품을 저장할 때: RDB에 원본을 쓰고(진실), 그 내용을 ES 인덱스에도 색인한다(검색용). 주문·결제 같은 정합성은 RDB가, 검색·추천은 ES가 맡는다.

</details>

## 동적 매핑 vs 명시적 매핑

- **동적 매핑** — 매핑을 안 만들고 그냥 문서를 색인하면, ES가 값을 보고 타입을 **자동 추정**한다. 편하지만, 문자열을 무조건 기본 분석기로 잡아서 **한글이 제대로 안 쪼개진다.**
- **명시적 매핑** — 인덱스를 만들 때 필드 타입을 직접 지정한다. 상품 검색은 **반드시 이 방식**으로, 상품명에 한글 분석기(nori)를 지정해야 한다.

<details>
<summary><b>🔍 동적 매핑·명시적 매핑을 실제로 어떻게 구현해?</b></summary>

**동적 매핑** — 매핑을 안 만들고 그냥 문서부터 넣으면 끝. 인덱스와 매핑이 자동 생성된다.

```json
POST products/_doc
{ "name": "무선 이어폰", "price": 39000, "onSale": true }
```

ES가 추론: `name` → `text` + `name.keyword`(멀티필드), `price` → `long`, `onSale` → `boolean`.
결과는 `GET products/_mapping`으로 확인한다.

**명시적 매핑** — 인덱스를 만들 때 `mappings`로 타입을 직접 지정.

```json
PUT products
{
  "mappings": {
    "properties": {
      "name":  { "type": "text", "analyzer": "nori",
                 "fields": { "keyword": { "type": "keyword" } } },
      "price": { "type": "integer" }
    }
  }
}
```

**섞어 쓰기(실무 권장)** — 핵심 필드는 명시적으로 잡고, 예상 못 한 새 필드가 들어올 때의 정책을 `dynamic`으로 정한다.

- `dynamic: "true"`(기본) — 모르는 새 필드가 오면 자동 추가
- `dynamic: "false"` — 저장은 하되 색인 안 함(검색 불가)
- `dynamic: "strict"` — 모르는 필드가 오면 **에러로 거부** (스키마 엄격 통제)

```json
PUT products
{ "mappings": { "dynamic": "strict", "properties": { /* ... */ } } }
```

상품 인덱스는 보통 **핵심 필드 명시적 + `dynamic: "strict"`**로 잡아, 오타 필드 하나가 매핑을 오염시키지 않게 한다.

</details>

## text vs keyword — 상품 검색의 핵심

문자열 필드는 두 타입 중 하나로 잡는다. **무엇을 고르냐가 검색 품질을 좌우한다.**

| | `text` | `keyword` |
|---|---|---|
| 처리 방식 | 분석기로 단어 단위로 **쪼갬** | 값 전체를 **통째로 하나**로 |
| 용도 | 전문 검색 ("무선"으로 "무선 이어폰" 찾기) | 정확 매칭·필터·집계·정렬 |
| 정렬·집계 | 기본 불가(비용 큼) | 가능 |
| 상품 예시 | 상품명, 설명 | 카테고리, 태그, 셀러ID, 상태 |

핵심 규칙: **"사람이 자유롭게 검색할 텍스트" = `text`, "값이 정해져 있고 거르거나 셀 것" = `keyword`.**
상품명은 `text`(무선으로도 걸려야 하니까), 카테고리는 `keyword`("전자기기"로 정확히 거르고 개수를 세니까).

## multi-field — 한 필드를 둘 다

상품명은 검색도 되어야 하고(text) 정렬·집계도 필요할 때가 있다(keyword). ES는 **한 필드를 여러 타입으로 동시에** 잡을 수 있다. 이게 `fields`(multi-field)다.

```json
"name": {
  "type": "text",
  "fields": {
    "keyword": { "type": "keyword", "ignore_above": 256 }
  }
}
```

이러면 검색은 `name`(text)으로, 정렬·집계는 `name.keyword`로 쓴다. **동적 매핑이 문자열에 이 구조를 자동으로 만들어준다** — 그래서 예제에서 `.keyword`를 자주 본다. (`ignore_above: 256`은 256자 넘는 값은 keyword로 색인 안 한다는 뜻 — 긴 텍스트가 집계에 안 끼게.)

## 자주 쓰는 나머지 타입

- **숫자** — `integer`, `long`, `float`, `double` 등. 가격·재고·개수.
- **날짜** — `date`. 등록일·수정일. 범위 검색과 날짜 히스토그램(트렌드 분석)에 쓴다.
- **불리언** — `boolean`. 판매중 여부 등.
- **object** — JSON 중첩 객체. 그냥 `{ "seller": { "id": ..., "name": ... } }`처럼 넣으면 ES가 내부적으로 `seller.id`, `seller.name`으로 펼쳐 저장한다.
- **nested** — object 배열을 다룰 때 필요한 특수 타입(아래에서).

## object vs nested — 배열 객체의 함정

상품 옵션처럼 **객체가 배열로 여러 개** 있을 때 조심해야 한다.

```json
"options": [
  { "color": "red",  "stock": 0 },
  { "color": "blue", "stock": 5 }
]
```

이걸 기본 `object`로 두면 ES는 내부적으로 각 필드를 납작하게 펴서 `color: [red, blue]`, `stock: [0, 5]`처럼 **따로따로** 저장한다. 그러면 "color=red **이면서** stock>0" 검색이 **잘못 걸린다** — red의 stock은 0인데, 배열 전체로 보면 stock에 5도 있으니 매칭돼 버린다.

**`nested` 타입**은 배열의 각 객체를 **독립된 문서처럼** 취급해서 이 문제를 없앤다. 대신 검색할 때 `nested` 쿼리를 써야 한다. "특정 옵션 조합"으로 검색·필터할 거면 nested로 잡는다.

## 매핑은 못 바꾼다 — 그래서 처음이 중요

**이미 만든 필드의 타입은 변경할 수 없다.** `text`로 잡은 걸 `keyword`로 바꾸거나, 분석기를 교체하는 건 불가능하다. 새 필드를 **추가**하는 건 되지만, 기존 필드를 고치려면 **새 인덱스를 만들어 데이터를 옮겨 담아야 한다(재색인)** — 1번 노트에서 본 그 재색인이다.

그래서 상품 인덱스 매핑은 처음에 신중히 잡고, 바꿀 일이 생기면 **별칭(alias) + 재색인**으로 무중단 교체한다(별칭은 뒤 노트에서).

## 상품 인덱스 스키마 설계 (종합)

지금까지를 상품 인덱스 하나로 모으면 이렇게 된다. 이게 이후 검색·집계·추천의 바탕이 된다.

```json
PUT products
{
  "mappings": {
    "properties": {
      "name":        { "type": "text", "analyzer": "nori",
                       "fields": { "keyword": { "type": "keyword", "ignore_above": 256 } } },
      "description": { "type": "text", "analyzer": "nori" },
      "price":       { "type": "integer" },
      "category":    { "type": "keyword" },
      "tags":        { "type": "keyword" },
      "sellerId":    { "type": "keyword" },
      "createdAt":   { "type": "date" },
      "onSale":      { "type": "boolean" },
      "options":     { "type": "nested",
                       "properties": {
                         "color": { "type": "keyword" },
                         "stock": { "type": "integer" } } }
    }
  }
}
```

- `name` — 검색은 text(nori로 한글 분석), 정렬·집계는 `name.keyword`
- `category`·`tags`·`sellerId` — 거르고 세야 하니 keyword
- `price`·`createdAt` — 범위 검색·정렬용 숫자/날짜
- `options` — 옵션 조합 검색을 위해 nested

`analyzer: "nori"`가 상품명 검색 품질의 핵심인데, 이 분석기가 정확히 뭘 하는지가 **3번 노트**의 주제다.
