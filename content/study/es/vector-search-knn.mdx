시리즈 마지막이다. 지금까지는 **단어(term)로** 검색했다(lexical). 이 노트는 **의미로** 검색·추천하는 벡터다. 8번 노트에서 "세어서 하는 추천" 다음이라 한 그 "의미로 하는 추천"이다.

## 벡터 검색이란

단어 매칭(BM25)과 **다른 축**이다. 텍스트를 **의미 좌표(벡터)**로 바꿔서, **좌표가 가까운 = 뜻이 비슷한** 것을 찾는다.

- "무선 이어폰"과 "블루투스 이어폰"은 단어가 달라도 좌표가 가깝다 → 벡터로는 유사.
- 이 "가까운 좌표 k개 찾기"가 **kNN**(5번 노트에서 개념, 여기선 문법).

핵심: 벡터는 **단어가 안 겹쳐도 의미로** 매칭한다. lexical이 못 하는 걸 채운다(뒤에서 언제 갈리나 정리).

<details>
<summary><b>🔍 lexical이 뭐야?</b></summary>

**단어(글자) 그대로 매칭하는 검색**이다. match·term·bool·BM25·nori가 전부 lexical — 실제 단어(term)가 겹치는지로 찾는다. 반대가 **semantic(의미)** = 벡터. "무선"으로 "무선 이어폰" 찾기가 lexical, "블루투스 이어폰"까지 의미로 찾기가 semantic이다.

</details>

## dense_vector 필드

벡터를 담는 필드 타입이다. 상품 매핑에 추가한다.

```json
PUT products
{
  "mappings": {
    "properties": {
      "name":      { "type": "text", "analyzer": "nori" },
      "embedding": { "type": "dense_vector", "dims": 768, "similarity": "cosine" }
    }
  }
}
```

- **`dims`** — 벡터 길이. 임베딩 모델이 정한다(예: 768).
- **`similarity`** — 가까움을 재는 방식. `cosine`(기본)·`dot_product`·`l2_norm` 등.
- kNN·dense_vector는 **ES 9.x 코어 내장**이다(nori와 달리 플러그인 불필요).

<details>
<summary><b>🔍 dims랑 similarity는 뭘 정하는 거야?</b></summary>

- **`dims`** — 벡터의 **길이(차원 수)**. 임베딩 모델이 정한다(768이면 숫자 768개짜리 리스트). ES는 미리 알아야 색인하므로 매핑에 못 박는다. 차원이 바뀌면 재색인 필요.
- **`similarity`** — 두 벡터가 얼마나 가까운지 재는 방식. `cosine`(기본, 텍스트 임베딩에 흔함)·`dot_product`(정규화했으면 빠름)·`l2_norm`(직선 거리). 보통 모델이 권하는 방식을 따른다.

</details>

## kNN 검색

내가 준 벡터와 가까운 문서 k개를 찾는다.

```json
POST products/_search
{
  "knn": {
    "field": "embedding",
    "query_vector": [0.7, 0.1, 0.9, ...],
    "k": 10,
    "num_candidates": 100
  }
}
```

- **`query_vector`** — **외부 모델이 만든** 유저/질의 벡터를 넣는 자리.
- **`k`** — 가까운 것 몇 개.
- **`num_candidates`** — 각 샤드에서 살펴볼 후보 수. **키우면 정확도 ↑, 속도 ↓**. ES는 전부 비교하는 대신 근사(approximate) kNN으로 빠르게 찾는다.

<details>
<summary><b>🔍 num_candidates는 뭐야? k랑 뭐가 달라?</b></summary>

ES는 수백만 벡터를 전부 비교하지 않고 근사(approximate)로 찾는다.

- **`k`** — 최종으로 **돌려받을** 개수(예: 10).
- **`num_candidates`** — 각 샤드에서 **후보로 살펴볼** 수(예: 100). 이 중 최종 k개를 추림.

`num_candidates`를 **키우면** 진짜 가까운 걸 놓칠 확률이 줄어 **정확도 ↑, 속도 ↓**(6번 `shard_size`와 같은 결). k는 결과 수, num_candidates는 정확도 손잡이.

</details>

## 벡터는 어디서 오나 — 역할 분리

여기가 8번 노트의 "역할 분리"와 직결된다.

- **임베딩 모델**(텍스트·이미지 → 벡터) = **ES 밖**. 무거운 ML. 상품을 색인할 때 미리 벡터를 만들어 `embedding` 필드에 저장.
- **ES** = 그 벡터를 **저장하고, kNN으로 가까운 걸 찾는** 것.

```
임베딩 모델(ES 밖) ─벡터 생성→ 상품 문서의 embedding 필드에 저장
검색 시: 유저/질의 벡터(모델이 생성) ─knn 쿼리→ ES가 가까운 상품 반환
```

즉 **벡터를 "만드는" 건 모델, "찾는(kNN)" 건 ES.** 모델을 바꿔도 ES kNN은 그대로다.

<details>
<summary><b>🔍 알고리즘은 ES 밖으로 뺐다면서, 왜 벡터는 ES에 저장해?</b></summary>

두 개를 나눠 봐야 한다.

- **벡터를 "계산"하는 것**(무거운 ML) = **모델, ES 밖.** 텍스트·이미지·행동 → 좌표(벡터)로 변환하는 계산. 이게 무거워서 뺐다.
- **벡터(결과물)를 "저장·검색"하는 것** = **ES.** 벡터는 그냥 **숫자 리스트(데이터)**다. ES에 넣어둬야 kNN으로 빠르게 찾는다.

비유: 모델 = 좌표를 **계산하는 계산기**(주방 요리사, 무거움), 벡터 = **완성된 좌표**(요리), ES = 좌표를 **진열해두고 가까운 걸 찾아주는 매대**. 요리사(모델)는 주방(밖)에 있지만 완성된 요리(벡터)는 매대(ES)에 올려둔다. **알고리즘(계산)을 뺀 거지 결과 데이터(벡터)를 뺀 게 아니다** — 오히려 ES에 넣어둬야 검색된다.

</details>

## more_like_this vs 벡터 — 언제 갈리나

8번 대화에서 정리한 것을 문법으로. **"비슷한 상품"은 두 방식**이 있다.

- **`more_like_this`(lexical)** — 그 상품의 태그·이름·설명 텍스트로 비슷한 문서를 찾는다. 태그가 잘 붙어 있으면 싸고 빠르고 충분하다.

```json
{ "query": { "more_like_this": {
    "fields": ["name", "description", "tags"],
    "like": [ { "_index": "products", "_id": "p456" } ],
    "min_term_freq": 1
} } }
```

- **벡터(kNN)** — 태그·단어가 표현 못 하는 의미·느낌 유사, 또는 "유저 취향처럼 비교할 문서가 없는" 경우.

정리(8번과 동일): **태그로 충분하면 more_like_this, 태그 너머 의미·취향이면 벡터.** 벡터가 더 정확하거나 가벼운 게 아니라, **다른 문제를 푼다.**

<details>
<summary><b>🔍 more_like_this는 어떻게 동작해?</b></summary>

"이 문서와 비슷한 문서"를 **lexical하게** 찾는다. 기준 문서(`like`)의 텍스트에서 **대표 단어들을 뽑아**, 그 단어로 다른 문서를 검색한다. 벡터가 아니라 **공유 단어 기반**이라 태그·이름·설명이 잘 되어 있으면 싸고 빠르다.

- `fields` — 어디서 비슷함을 볼지(name·description·tags)
- `like` — 기준(문서 id 또는 임의 텍스트)
- `min_term_freq`·`min_doc_freq` — 너무 드물거나 흔한 단어를 걸러 대표 단어 질 조절

태그가 못 잡는 의미·취향 유사가 필요하면 벡터로 간다.

</details>

## 하이브리드 — lexical + 벡터 합치기

둘 다의 장점을 원할 때(정확한 단어 매칭 + 의미 확장), **한 요청에 `query`와 `knn`을 같이** 준다.

```json
POST products/_search
{
  "query": { "match": { "name": "무선 이어폰" } },       // lexical (BM25)
  "knn":   { "field": "embedding", "query_vector": [...],
             "k": 10, "num_candidates": 100 }             // 의미 (벡터)
}
```

- 두 점수를 **가중 합산**한다: `점수 = query_boost×BM25 + knn_boost×벡터`. 각 `boost`로 비중을 조절.
- 순위 성격이 다른 둘을 더 안정적으로 섞으려면 **RRF(Reciprocal Rank Fusion)**를 쓴다 — 점수가 아니라 **각 방식의 순위를 융합**해 한 랭킹으로. (9.x의 retrievers로 구성; 세부는 공식 문서 참고.)

하이브리드는 커머스 검색 품질의 상한을 올리지만, **순수 키워드 검색만이면 lexical로 충분**할 때가 많다(8번 결론).

<details>
<summary><b>🔍 하이브리드 "가중 합산"이 정확히 뭐야?</b></summary>

한 요청에 `query`(BM25)와 `knn`(벡터)를 같이 주면, 각 문서 최종 점수를 **두 점수의 가중 합**으로 낸다:

`최종점수 = query_boost × BM25점수 + knn_boost × 벡터점수`

각 `boost`로 비중을 조절한다("단어 매칭 중시" vs "의미 중시"). 단, BM25와 벡터는 스케일이 달라 그냥 더하면 한쪽이 지배할 수 있다 — 그래서 순위로 합치는 **RRF**를 쓰기도 한다(아래 RRF 토글).

</details>

<details>
<summary><b>🔍 RRF(Reciprocal Rank Fusion)가 뭐야?</b></summary>

두 개의 다른 순위표를 공정하게 합치는 규칙이다. 하이브리드에서 BM25 순위 + 벡터 순위를 섞을 때, **점수를 그냥 더하면** BM25(예: 12.5)와 벡터(예: 0.87)의 **스케일이 달라** 한쪽이 지배한다. RRF는 점수 대신 **"몇 등이냐(순위)"만 보고** 합친다(대략 `1/(k+순위)`를 더함 → 위에 있을수록 기여 큼). 그래서 스케일 문제 없이 두 검색을 병합한다. (9.x의 retrievers로 구성.)

</details>

## 상품 추천·시맨틱 검색에 어떻게 (종합)

```
임베딩 모델(밖) → 상품·유저 벡터 → ES embedding 필드
검색/추천:
  ├ 키워드 검색      → match/bool + BM25 (2~4번)
  ├ 비슷한 상품(태그) → more_like_this
  ├ 의미·취향 유사    → knn (이 노트)
  └ 품질 상한 올리기  → 하이브리드(query + knn, RRF)
+ function_score로 개인화·인기 얹기(8번) · 결과는 productId로, 이름은 나중(8번)
```

- **당신** = 벡터를 ES에 저장·조회(kNN)하고, lexical과 언제 섞을지 설계.
- **모델 담당** = 임베딩 생성.

## 시리즈를 마치며

1~9번으로 "한글 상품 검색 + 행동 로그 분석 + 추천"의 ES 설계 뼈대가 섰다: 역색인·샤드(1) → 매핑(2) → nori 분석기(3) → 쿼리·랭킹(4) → 검색어 제안(5) → 집계 분석(6) → 이벤트 파이프라인(7) → 추천 설계(8) → 벡터·하이브리드(9). 이제 이걸 **로컬 ES에서 검증하고 Prompthub에 실제 구현**하면, 그 과정이 다시 project·troubleshooting·decision 콘텐츠가 된다.
