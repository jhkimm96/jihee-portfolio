2번 노트에서 상품명을 `text`로 잡고 `analyzer: "nori"`를 붙였다. 그 nori가 뭘 하는지 — **text가 실제로 어떻게 검색 가능한 단어로 쪼개지는지**가 이 노트다. 여기가 한글 상품 검색 품질의 핵심이다.

## 분석기란

**분석기(analyzer)**는 `text` 필드의 문자열을 **검색 가능한 단어(term)로 쪼개고 다듬는** 장치다. 1번 노트의 역색인이 "단어 → 문서" 표라면, 그 "단어"를 만들어내는 게 분석기다.

- **`text` 필드에만** 적용된다. `keyword`는 통째로 저장하므로 분석기를 안 거친다.
- 색인할 때 한 번(문서를 넣을 때), 검색할 때 한 번(검색어를 넣을 때) 돈다.

## 분석기의 3단계 구조

분석기는 세 단계를 순서대로 거친다.

```
원본 텍스트
  → ① 문자 필터(character filter)   문자 수준 전처리 (HTML 제거, 문자 치환)
  → ② 토크나이저(tokenizer)          텍스트를 토큰으로 쪼갬 ★핵심
  → ③ 토큰 필터(token filter)        토큰 후처리 (소문자화, 불용어 제거, 동의어)
  → 최종 term들 → 역색인에 저장
```

- **① 문자 필터** — 쪼개기 전에 문자를 손본다. 예: HTML 태그 제거, "&" → "and" 치환. 없어도 된다(0개 이상).
- **② 토크나이저** — 문자열을 토큰 단위로 자른다. **분석기당 정확히 1개.** 영어 기본(`standard`)은 공백·구두점 기준으로 자르고, 한글은 여기서 nori를 쓴다.
- **③ 토큰 필터** — 잘린 토큰을 다듬는다. 소문자화(`lowercase`), 불용어 제거(`stop`), 동의어(`synonym`), 한글 품사 제거(`nori_part_of_speech`) 등. 여러 개를 순서대로 건다.

## 색인 분석 vs 검색 분석 — 같아야 걸린다

핵심 함정: **색인할 때와 검색할 때 같은 방식으로 쪼개져야 매칭된다.**

색인 때 "무선이어폰"을 `무선`, `이어폰`으로 쪼갰는데, 검색 때 "이어폰"이 안 쪼개지거나 다른 분석기를 쓰면 term이 안 맞아 검색이 안 된다. 그래서 보통 **색인·검색에 같은 분석기**를 쓴다.

- `analyzer` — 색인·검색 둘 다에 쓸 분석기(기본).
- `search_analyzer` — 검색 때만 다르게 쓰고 싶을 때 따로 지정(예: 자동완성에서 쓰는 고급 기법. 5번 노트에서).

<details>
<summary><b>🔍 색인·검색 분석이 안 맞으면 구체적으로 뭐가 깨져? search_analyzer는 언제?</b></summary>

색인 때 "무선이어폰"을 mixed nori로 넣으면 term은 `무선이어폰`,`무선`,`이어폰`이 된다. 그런데 검색 때 다른 규칙으로 쪼개면(예: standard) 검색어 "이어폰"이 그 term들과 안 맞아 **결과 0건**이 된다. 색인·검색이 같은 규칙으로 쪼개져야 term이 겹친다.

`search_analyzer`를 따로 쓰는 대표 사례는 **동의어**와 **자동완성**이다. 동의어를 검색 때만 확장하거나, 자동완성(edge n-gram)에서 색인은 잘게·검색은 통째로 넣는 식으로 분리한다(5번 노트).

</details>

## 왜 한글은 형태소 분석이 필요한가

영어 기본 분석기(`standard`)는 **공백·구두점 기준**으로 자른다. 영어는 단어가 공백으로 갈려서 이걸로 충분하다. 하지만 한글은:

- 조사가 붙는다: "이어폰**을**", "무선**으로**" → `이어폰을`, `무선으로`가 통째로 term이 됨 → "이어폰"으로 검색해도 안 걸림.
- 붙여쓰기: "무선이어폰" → 통째로 한 term → "이어폰"으로 안 걸림.

즉 standard로는 한글 상품명 검색이 사실상 망가진다. **의미 단위(형태소)로 쪼개는** 도구가 필요하고, 그게 nori다.

## nori — 공식 한글 형태소 분석기

nori는 Elasticsearch **공식 한글 형태소 분석 플러그인**이다.

- **설치가 필요하다.** self-managed는 모든 노드에 `bin/elasticsearch-plugin install analysis-nori`로 깔아야 한다(클러스터 전 노드 필수). Elastic Cloud/서버리스는 기본 포함.
- **내장 `nori` 분석기**를 바로 쓸 수 있다 — 2번 노트에서 `analyzer: "nori"`라고 쓴 게 이것.
- 세밀 조정이 필요하면 아래처럼 **커스텀 분석기**를 만든다.

### 핵심 구성요소

- **`nori_tokenizer`** — 한글을 형태소로 쪼갠다. 복합어 처리를 `decompound_mode`로 정한다:
  - `none` — 복합어를 안 쪼갬 (`가곡역` → `가곡역`)
  - `discard` (기본) — 복합어를 쪼개고 원형은 버림 (`가곡역` → `가곡`, `역`)
  - `mixed` — 쪼갠 것 + 원형 둘 다 (`가곡역` → `가곡역`, `가곡`, `역`)
- **`nori_part_of_speech`** (토큰 필터) — 품사 기준으로 불필요한 토큰(조사·어미 등)을 걸러낸다. "이어폰을"에서 조사 "을"을 버려 `이어폰`만 남긴다.
- **`nori_readingform`** (토큰 필터) — 한자를 한글 읽기로 바꾼다(한자 상품명 대비).

**상품 검색 팁**: 상품명은 `mixed`가 무난하다. "무선이어폰"이 `무선이어폰`(원형) + `무선` + `이어폰`으로 다 색인돼, 어느 쪽으로 검색해도 걸린다.

<details>
<summary><b>🔍 decompound_mode를 바꾸면 검색이 어떻게 달라져?</b></summary>

"무선이어폰"이 각 모드에서 만드는 term:

- `none` → `무선이어폰`만. "이어폰"으로 검색하면 **안 걸림.**
- `discard` → `무선`, `이어폰`. "이어폰"으로 걸리지만, 붙여쓴 원형 "무선이어폰"은 term에 없음.
- `mixed` → `무선이어폰`, `무선`, `이어폰`. "이어폰"으로도, "무선이어폰"으로도 걸림.

그래서 커머스 상품명은 보통 `mixed`가 안전하다 — 부분 단어와 원형을 모두 잡으니까. 대신 term이 늘어 색인이 약간 커진다.

</details>

## 커스텀 nori 분석기 만들기

`settings`의 `analysis` 블록에서 분석기를 정의하고, `mappings`에서 필드에 지정한다.

```json
PUT products
{
  "settings": {
    "analysis": {
      "tokenizer": {
        "nori_mixed": { "type": "nori_tokenizer", "decompound_mode": "mixed" }
      },
      "analyzer": {
        "korean": {
          "type": "custom",
          "tokenizer": "nori_mixed",
          "filter": ["nori_part_of_speech", "lowercase"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "name": {
        "type": "text",
        "analyzer": "korean",
        "fields": { "keyword": { "type": "keyword", "ignore_above": 256 } }
      }
    }
  }
}
```

이러면 상품명이 `korean` 분석기로 처리된다: nori로 형태소 분해(mixed) → 조사·어미 제거 → 소문자화.

<details>
<summary><b>🔍 이 PUT products는 문서인가? RDB 테이블/컬럼이랑은 무슨 관계?</b></summary>

**ES와 RDB는 별개 저장소다.** 이 매핑은 RDB product 테이블에 컬럼을 추가하는 게 아니라, ES 안에 검색용 `products` 인덱스를 따로 만드는 것이다. 대응 관계:

| RDB | ES |
|---|---|
| 테이블 `product` | 인덱스 `products` |
| 행(상품 한 건) | 문서(document) |
| 컬럼 `name`·`price` | 필드(field) |
| `CREATE TABLE` 스키마 | 매핑(mapping) |

- 이 `PUT products` JSON = 인덱스 **틀(구조 정의)**. RDB의 `CREATE TABLE`에 해당. 한 번 만든다.
- **`name`** = 상품명 필드. **`fields.keyword`** = 그 상품명을 정렬·집계용으로도 한 벌 더 저장(2번 노트 multi-field).
- **못 바뀌는 건 "틀"(매핑·분석기).** 문서(상품 데이터)는 자유롭게 추가·수정·삭제된다.
- **기존 문서는?** `PUT products` 직후엔 문서 0개(빈 틀). 이후 상품을 하나씩 문서로 넣으면 인덱스에 쌓인다:

```json
PUT products { "settings": { /*분석기*/ }, "mappings": { /*필드*/ } }   // 틀 (한 번)
POST products/_doc { "name": "무선 이어폰", "price": 39000 }            // 상품 문서 (계속)
```

분석기를 바꿔야 하면 틀은 못 고치니, 새 인덱스를 만들어 문서를 옮겨 담는다(재색인).

</details>

## 동의어 — 검색 품질을 크게 올림

사용자는 "블루투스 이어폰"이라 쳐도 "무선 이어폰" 상품이 나오길 기대한다. **동의어 필터(`synonym`)**로 같은 뜻 단어를 묶는다.

```json
"filter": {
  "product_synonyms": {
    "type": "synonym",
    "synonyms": ["무선, 블루투스", "노트북, 랩탑"]
  }
}
```

이 필터를 분석기의 `filter` 목록에 넣으면 "무선"과 "블루투스"가 같은 것으로 취급돼 서로 검색된다. 커머스 검색에서 체감 품질을 가장 크게 좌우하는 부분이다.

<details>
<summary><b>🔍 동의어는 색인 때 넣나 검색 때 넣나? (바꿀 때 재색인 함정)</b></summary>

동의어 필터는 두 곳에 넣을 수 있고 성격이 다르다.

- **색인 시 적용** — 색인할 때 동의어를 term으로 함께 저장. 검색은 빠르지만, **동의어 목록을 바꾸면 기존 문서를 재색인**해야 반영된다.
- **검색 시 적용**(`search_analyzer`에만) — 검색어에만 동의어 확장. **바꿔도 재색인 불필요**, 바로 반영. 대신 검색 때 살짝 비용.

동의어는 운영하며 자주 추가되므로 재색인 부담을 피하려 **검색 시 적용**을 많이 쓴다. 또 "무선 이어폰"처럼 여러 단어 동의어를 안전히 다루려면 `synonym` 대신 `synonym_graph`를 쓴다.

</details>

## _analyze로 직접 확인

분석기가 실제로 어떻게 쪼개는지 API로 바로 테스트할 수 있다. 설계할 때 필수 도구다.

```json
POST products/_analyze
{ "analyzer": "korean", "text": "무선이어폰을 찾아요" }
```

응답으로 `무선이어폰`, `무선`, `이어폰`, `찾` 같은 최종 term 목록이 나온다. 기대와 다르면 `decompound_mode`나 필터를 조정한다.

<details>
<summary><b>🔍 _analyze, 인덱스 안 만들고도 테스트할 수 있어?</b></summary>

가능하다. 인덱스 없이 분석기를 즉석에서 정의해 바로 돌려볼 수 있다 — 설계 초기에 유용하다.

```json
POST _analyze
{
  "tokenizer": "nori_tokenizer",
  "filter": ["nori_part_of_speech"],
  "text": "무선이어폰을 찾아요"
}
```

응답의 각 토큰엔 `token`(단어), `position`(순서), `start_offset`/`end_offset`(원문 위치)이 있다. 토크나이저만, 또는 필터를 하나씩 넣었다 빼며 "어디서 이렇게 쪼개졌나"를 짚어 문제를 찾는다.

</details>

## 상품 검색에 어떻게 쓰나 (종합)

- 상품명·설명 = `text` + **커스텀 nori 분석기**(mixed 분해 + 품사 필터 + 동의어)
- 정렬·집계용 = `name.keyword`(2번 노트)
- 배포 시 **모든 노드에 analysis-nori 설치** 확인
- 분석기 바꾸면 기존 문서엔 소급 안 되므로 **재색인** 필요(1·2번 노트) — 그래서 분석기도 처음에 신중히

분석기로 "무엇이 term이 되는가"가 정해졌으니, 다음은 그 term들을 **어떻게 검색하고 순위를 매기나** — 쿼리 DSL과 관련도(BM25)가 **4번 노트**다.
