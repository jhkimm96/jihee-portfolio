## 시작한 질문

파일(이미지·산출물)을 업로드할 때마다 서버 heap 사용량이 올라간다. "이미 올라간 heap을 어떻게 줄이지? GC 설정을 해야 제거되나?"에서 출발했다.

결론부터: **GC가 쓰레기를 치우는 건 설정이 필요 없는 자동 동작**이다. 헷갈리기 쉬운 지점을 하나씩 분리한다.

## 1. 힙에 올라가고, GC가 자동으로 치운다

업로드한 파일을 서버가 `byte[]`로 들고 있으면 heap 사용량이 올라간다. 요청이 끝나면 그 객체는 "쓰레기(dead object)"가 되고, **다음 GC 때 자동 회수**된다. 아무 설정도 필요 없다 — JVM의 기본 동작이다.

> 비유: 방에서 나온 쓰레기를 청소기가 알아서 치운다. 스위치를 따로 켤 필요가 없다.

<details>
<summary><b>🔍 "다음 GC 때"가 정확히 언제야? GC는 언제 도나</b></summary>

GC는 정해진 시각에 도는 게 아니라 **힙이 차면(할당 압력이 생기면)** 트리거된다. 시간 기반이 아니라 **양 기반**이다.

- 새 객체(`byte[]` 등)는 힙의 **Eden** 영역에 쌓인다.
- Eden이 꽉 차면 → **Young GC**가 자동 실행(자주·빠름)되고, 참조 끊긴 쓰레기는 그 순간 회수된다.
- 그래서 "다음 GC 때" = **다음번 Eden이 찰 때**. 업로드가 이어지면 보통 몇 초~수십 초 안.
- 트래픽이 없으면 Eden이 안 차서 GC가 한동안 안 일어날 수도 있다 — 유휴 시 메모리가 안 줄어 보이는 이유 중 하나. 필요가 없으니 안 치우는 것(효율적).

한 줄: **"메모리가 필요해지는 시점에 알아서 돈다."** 예약된 시각은 없다.

</details>

## 2. GC는 "쓰레기만" 치운다 — live 객체는 못 줄인다

중요한 전제: 줄일 수 있는 건 **쓰레기(더 이상 안 쓰는 객체)뿐**이다. 아직 참조가 살아있는(live) 객체는 GC가 못 건드린다.

- GC 돌렸는데 heap이 확 내려감 → 쓰레기가 쌓여있던 것 (정상)
- GC 돌려도 안 내려감 → 진짜 살아있는 데이터거나 **메모리 누수** → GC로는 못 줄이고 코드를 봐야 함

## 3. "힙 안에서 회수" ≠ "OS에 반납"

함정. GC가 힙 안의 쓰레기를 치워도 **JVM은 그 빈 공간을 OS에 잘 안 돌려준다.** 그래서:

- 힙 내부는 비었는데
- `docker stats`/RSS(OS가 보는 메모리)는 계속 높게 보인다

이 "유휴 메모리를 OS에 반납(uncommit)"을 유도하는 게 `-XX:G1PeriodicGCInterval` 같은 옵션이다. 유휴 시간에 주기적 GC를 돌려 안 쓰는 영역을 OS에 돌려준다. **성능/안정성이 아니라 "겉보기 메모리 숫자"를 개선하는 용도**다.

| 하고 싶은 것 | 방법 |
|---|---|
| 힙 used 줄이기 | GC (자동) |
| RSS(겉보기 메모리) 줄이기 | OS 반납 성향 GC 옵션 |

## 4. GC vs 재시작 — 재시작은 청소가 아니라 리셋

가장 헷갈리는 부분. 컨테이너를 재시작하면 메모리가 확 주는데, 그건 **GC가 청소해서가 아니다.**

| | GC (가비지 컬렉션) | 재시작 |
|---|---|---|
| 언제 | 앱 도는 중 자동 | 프로세스 죽이고 새로 |
| 방식 | 쓰레기만 골라 회수, live는 유지 | 힙 통째로 파괴 후 빈 상태로 재생성 |
| 비유 | 쓰레기만 골라 버리는 청소 | 방을 비우고 새 방으로 이사 |

재시작으로 메모리가 준 건 **JVM 프로세스 자체가 사라지며 그 메모리 전체를 OS가 회수**하고 새 JVM이 빈 힙으로 떠서다. GC처럼 똑똑하게 고르는 게 아니라 **필요한 것(캐시·진행 중 작업)까지 다 날리는 리셋**이라 downtime도 생긴다.

## 5. 실무: 이미 올라간 heap 줄이는 방법

무해한 것부터 과감한 것 순으로:

1. **GC 강제 실행 (재시작 없이)** — `jcmd <pid> GC.run`. 쓰레기 회수 + (JDK 12+ G1은) 안 쓰는 영역 OS 반납.
   - ⚠️ **JRE(슬림) 이미지엔 `jcmd`/`jstat`이 없다.** 진단 도구는 JDK에만 들어있어, 배포용 JRE 이미지에선 이 명령이 `executable file not found`로 안 먹는다. 그럴 땐 아래로.
2. **컨테이너 재시작** — 확실한 리셋. 도구 불필요, 대신 잠깐 downtime.
3. **상한 걸기(예방)** — `-XX:MaxRAMPercentage` / `-Xmx`로 천장 고정. 이미 올라간 걸 줄이는 게 아니라 그 이상 못 커지게.

<details>
<summary><b>🔍 MaxRAMPercentage가 정확히 뭘 하나</b></summary>

JVM이 쓸 수 있는 **최대 heap 크기를 컨테이너 메모리의 비율(%)로** 지정한다.

- 예: 컨테이너 4GB + `MaxRAMPercentage=25` → 최대 heap ≈ 1GB.
- **왜 %?** `-Xmx1g` 고정값은 컨테이너 메모리를 늘려도 안 따라온다. %면 컨테이너 메모리에 비례해 자동 조정된다.
- **막는 것**: heap 천장을 고정해 한 서비스가 메모리를 무한정 먹어 다른 컨테이너를 밀어내는 걸 예방(이미 올라간 걸 줄이는 게 아니라 천장 설정).
- ⚠️ 컨테이너에 메모리 limit을 안 걸면 JVM이 **호스트 전체 메모리** 기준으로 %를 계산한다 → 예상보다 큰 heap.

</details>

## 6. 근본책: 애초에 힙에 안 올리기

heap이 오르는 것 자체가 문제라면 GC 튜닝으론 못 푼다(큰 파일을 heap에 올리는 한 OOM을 늦출 뿐). 진짜 해결은 **파일 바이트를 서버 heap에 안 올리는 구조**다 — 파일 업로드를 [presigned PUT으로 전환](/decisions/prompthub/product-service/presigned-put-upload-heap-fix)한 결정이 그 예다.
