## MSA란?

MSA(Microservice Architecture)는 큰 애플리케이션을 여러 개의 작은 독립 서비스로 쪼갠 구조다. 반대는 모놀리식(Monolith)으로, 모든 기능이 하나의 큰 덩어리 안에 있고 한 번에 배포된다.

**비유 — 1인 식당 vs 역할 나눈 식당**

- 모놀리식 = 주문·요리·계산·설거지를 혼자 하는 1인 식당. 작을 땐 편하지만 바빠지면 그 한 명이 병목이고, 요리사가 아프면 가게 전체가 멈춘다.
- MSA = 웨이터·요리사·계산원이 각각 독립된 구조. 필요한 파트만 인원을 늘릴 수 있고, 한 파트가 멈춰도 나머지는 돌아간다.

## 왜 쪼개나 (장점)

- **독립 배포**: 상품 기능만 고치면 상품 서비스만 다시 배포한다.
- **독립 확장**: 부하가 몰리는 서비스만 늘린다.
- **장애 격리**: 결제가 죽어도 상품 조회는 된다.
- **팀 분업**: 팀마다 서비스를 하나씩 맡아 독립 개발한다.

### 독립 확장이란

특정 서비스만 똑같이 여러 개(인스턴스) 실행해 부하를 나눠 받는 것이다. 주문이 몰리면 order-service를 여러 개 띄우고(order-1, order-2, order-3), 로드밸런서가 요청을 나눠준다. 상품 서비스는 그대로 둔다 — 그래서 "독립" 확장이다. 실제로는 도커 이미지를 여러 개 실행하고 쿠버네티스가 개수를 자동 조절(오토스케일링)한다. 모놀리식이면 전체가 한 덩어리라 원하는 부분만 못 늘리고 통째로 복제해야 한다.

## 단점 (트레이드오프)

- **복잡함**: 서비스끼리 통신해야 한다(gRPC/Kafka/REST).
- **분산 트랜잭션이 어렵다**: SAGA 패턴으로 해결한다(아래 토글).
- **운영 부담**: 서비스가 많으면 배포·모니터링도 그만큼 늘어난다.
- **네트워크 지연**: 서비스 간 호출은 네트워크를 건너간다. gRPC로 줄일 수는 있어도 0은 아니며, 모놀리식의 함수 호출(메모리 안)보다는 여전히 느리다.

## PromptHub 서비스 지도

**비즈니스 서비스**

| 서비스 | 하는 일 |
| --- | --- |
| product-service | 상품(등록·조회·리뷰·이미지) |
| order-service | 주문·장바구니 |
| payment-service | 결제 |
| user-service | 사용자·판매자 |
| settlement-service | 정산 |
| admin-service | 관리자 |

**인프라 서비스(조연)**

| 서비스 | 역할 |
| --- | --- |
| apigateway | 정문 — 모든 요청이 먼저 도착(인증·라우팅) |
| discovery(Eureka) | 서비스 위치를 찾는 전화번호부 |
| config | 설정 중앙 관리 |
| common-module | 여러 서비스가 공유하는 공통 코드 |

## 실제로 어떻게 통신하나

| 통신 | 프로토콜 | 용도 |
| --- | --- | --- |
| gRPC | HTTP/2 (바이너리) | 내부 서비스 간 동기 호출 |
| REST | HTTP/1.1 (JSON) | 브라우저 상대 외부 API |
| Kafka | 자체 TCP 프로토콜 | 비동기 이벤트 |

내부는 성능을 위해 gRPC, 외부는 호환성·편의를 위해 REST, 비동기 처리는 Kafka를 쓴다. PromptHub에서 gRPC 서버는 product·order·user에 있어 그 내부 통신은 HTTP/2를 자동으로 쓴다. 외부 REST는 별도 설정이 없어 HTTP/1.1이다.

## 요청 흐름 예시 (상품 구매)

```
브라우저
  → apigateway (로그인 확인)
     → product-service (상품 조회)
  주문 → order-service → payment-service (결제)
                       → product-service (판매수 반영, Kafka)
```

<details>
<summary>💡 SAGA 패턴 (분산 트랜잭션)</summary>

모놀리식은 "주문+결제+재고차감"을 하나의 트랜잭션으로 묶어 다 되거나 다 롤백한다. MSA는 서비스마다 DB가 달라 하나로 못 묶는다. 그래서 "결제는 됐는데 재고 차감 실패" 같은 불일치가 생길 수 있다.

SAGA는 작업을 단계로 나누고, 중간에 실패하면 앞 단계를 되돌리는(보상 트랜잭션) 방식이다. 재고 실패 시 결제를 환불하고 주문을 취소해 일관성을 복구한다. 이벤트로 각자 처리하는 Choreography 방식과, 조정자가 순서를 지휘하는 Orchestration 방식이 있다.

</details>

<details>
<summary>💡 gRPC는 왜 빠를까? (바이너리 + HTTP/2)</summary>

- **바이너리**: REST는 사람이 읽는 텍스트(JSON)라 크고 무겁다. gRPC는 0/1로 압축한 바이너리(Protobuf)라 작고 파싱이 빠르다.
- **HTTP/2**: HTTP/1.1은 요청을 한 줄로 하나씩 처리한다(1차선 도로). HTTP/2는 한 연결에서 여러 요청을 동시에 처리한다(여러 차선). gRPC는 HTTP/2를 필수로 쓴다.

작은 데이터 + 여러 요청 동시 = gRPC가 내부 통신에서 빠른 이유다.

</details>

<details>
<summary>💡 HTTP 버전 역사</summary>

| 버전 | 나온 해 | 특징 |
| --- | --- | --- |
| HTTP/1.0 | 1996 | 초창기 |
| HTTP/1.1 | 1997 | 가장 오래·널리 쓰임(아직도 많이 씀) |
| HTTP/2 | 2015 | 여러 요청 동시(멀티플렉싱), 바이너리 |
| HTTP/3 | 2022 | 최신(UDP 기반) |

HTTP/2는 2015년에 나온, 이미 자리잡은 표준이다. 일반 웹/REST는 여전히 1.1을 많이 쓰고, gRPC는 HTTP/2를 필수로 쓴다.

</details>
