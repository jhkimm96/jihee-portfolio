## 한 줄 질문

"우리 서비스에 Outbox가 있으면 Inbox도 무조건 있어야 하는 거 아냐?"

답: 아니다. **Outbox(발행측)와 Inbox(소비측)는 짝이 아니다.** 각각 다른 side의 다른 문제를 푼다.

## 두 패턴이 푸는 것

- **Outbox (발행측)**: DB 저장과 Kafka 발행을 원자적으로 못 묶는 dual-write 문제. 보낼 메시지를 같은 트랜잭션으로 DB에 적어두고 relay가 발행 → **유실 방지(at-least-once)**. 대신 중복 발행이 생길 수 있다.
- **Inbox (소비측)**: at-least-once라 같은 이벤트가 여러 번 온다. 소비 연산이 비멱등이면 중복이 치명적. `(eventId, consumerGroup)` 유니크로 처리 이력을 남겨 **중복 반영 방지**.

Outbox는 "내가 **보내는** 걸 보장", Inbox는 "내가 **받는** 걸 한 번만 반영". 방향이 달라서 한 서비스가 둘 다일 수도, 하나만일 수도, 없을 수도 있다.

## 우리 MSA의 실제 현황

| 서비스 | Outbox(발행) | Inbox(소비 멱등) | 이유 |
|---|---|---|---|
| order-service | ✅ | ✅ | 주문 이벤트를 유실 없이 발행 + 결제 이벤트를 비멱등 소비 |
| settlement-service | ✅ | ❌ | 정산 이벤트 발행만 |
| **product-service** | ❌ (직접 발행) | **✅** | order 이벤트로 salesCount를 증감 = 비멱등 |
| payment-service | ❌ | ❌ | order 이벤트를 소비하지만 연산이 멱등 |
| user-service | ❌ | ❌ | 정산 이벤트 소비, 누적 아님 |

**"Outbox만"(settlement) · "Inbox만"(product) · "둘 다"(order) · "둘 다 없음"(payment/user)** 이 전부 정상이다.

## 왜 product-service만 Inbox가 필요한가

핵심은 **소비 연산이 멱등이냐**다.

product-service는 `order-events`를 받아 판매수를 증감한다:

```java
// product-service — OrderEventHandler
@Transactional
public void handlePaid(UUID eventId, LocalDateTime occurredAt, List<UUID> productIds) {
    if (alreadyProcessed(eventId)) {   // ← Inbox: 이미 처리한 eventId면 skip
        return;
    }
    productSalesCountService.incrementSalesCount(productIds);   // salesCount += 1
    markProcessed(eventId, "ORDER_PAID", occurredAt);           // ← 처리 도장
}

private boolean alreadyProcessed(UUID eventId) {
    return processedEventRepository.existsByEventIdAndConsumerGroup(eventId, CONSUMER_GROUP);
}
```

`salesCount += 1`은 **여러 번 하면 결과가 달라지는(비멱등)** 연산이다. 같은 ORDER_PAID가 재전송되면 판매수가 +2 → 틀림. 그래서 `eventId`로 중복을 막는 Inbox가 **반드시** 필요하다.

<details>
<summary><b>🔍 멱등 / 비멱등이 뭔가</b></summary>

**멱등(idempotent)**: 같은 연산을 몇 번 하든 결과가 같은 것. 예) `상태 = PAID로 세팅` → 1번 하든 5번 하든 PAID. `id로 삭제` → 여러 번 해도 결과 같음.

**비멱등(non-idempotent)**: 다시 하면 결과가 달라지는 것. 예) `salesCount += 1` → 3번 하면 +3. `새 row INSERT` → 중복 생성.

왜 중요한가: Kafka는 at-least-once(같은 메시지가 여러 번 올 수 있음)라, 소비 연산이 **멱등이면 중복이 와도 무해**하지만 **비멱등이면 중복이 데이터를 망친다** → 그래서 비멱등 소비에만 Inbox(eventId 중복 제거)가 필요하다.

</details>

반면 payment-service는 같은 `order-events`를 받지만 하는 일이 다르다:

```java
// payment-service — OrderEventConsumer (ORDER_CREATED만 처리)
recordOrderSnapshotUseCase.record(new RecordOrderSnapshotCommand(
    created.orderId(),        // ← orderId 키로 스냅샷 기록
    created.buyerId(),
    created.totalAmount(),
    created.createdAt().atOffset(KST),
    OrderSnapshotSource.EVENT
));
```

orderId 키로 **고정된 값을 기록**하는 연산이라, 같은 이벤트가 두 번 와도 같은 스냅샷을 다시 쓸 뿐 결과가 안 바뀐다(**자연 멱등**). 그래서 Inbox 없이도 안전하다.

## 정리

- Outbox = 발행 유실 방지 / Inbox = 비멱등 소비 방지 — **독립적 선택**.
- Inbox는 **소비 연산이 멱등이 아닐 때만** 단다. product-service의 salesCount 증감이 바로 그 경우.
- "Outbox 있으니 Inbox도"가 아니라, **"내가 받는 이벤트로 하는 일이 누적/비멱등인가?"** 로 판단한다.
