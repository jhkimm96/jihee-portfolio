## 문제

상품 검색 색인 파이프라인(생성/수정 시 실시간 색인 + 7일 주기 전체 재조정 배치)을
배포했는데, ES `products` 인덱스에 문서가 단 한 건도 쌓이지 않았다. Kafka 컨슈머
로그를 보면 이벤트는 정상적으로 수신하고 있었지만, 처리 도중 재시도만 반복하다가
결국 DLT(Dead Letter Topic) 발행까지 실패하며 메시지가 그대로 유실되는 상태였다.

## 원인 조사 과정

Kafka 컨슈머의 기본 에러 핸들러(`DefaultErrorHandler`)는 재시도 중엔 실제 예외의
스택트레이스를 남기지 않고 "재시도 중" 요약 로그만 남긴다는 걸 몰라서, 처음엔
`grep "error|exception"`으로 로그를 뒤졌지만 무관한 에러(스케줄러 레이스 컨디션)만
잡혔다. 컨슈머가 처리하는 각 단계(멱등성 체크, DB 조회, 평균 평점·판매량 집계)를
SQL 로그로 하나씩 확인해 전부 정상 실행되는 걸 확인한 뒤, 남은 용의선상은 마지막
단계인 ES 색인 호출(`upsert`)뿐이라고 좁혔다. 그 다음 코드를 직접 다시 읽는 방식으로
전환해 `ElasticsearchClientConfig`의 `JacksonJsonpMapper` 생성부를 확인했고, 실제로
같은 시간대 로그를 넓게(시간 기준으로) 다시 받아서 진짜 스택트레이스를 잡아냈다.

```
jakarta.json.JsonException: Jackson exception
    at co.elastic.clients.json.jackson.JacksonJsonpMapper.serialize(...)
Caused by: com.fasterxml.jackson.databind.exc.InvalidDefinitionException:
Java 8 date/time type `java.time.LocalDateTime` not supported by default:
add Module "com.fasterxml.jackson.datatype:jackson-datatype-jsr310" to enable handling
(through reference chain: com.prompthub.search.infra.es.ProductSearchDocument["firstPublishedAt"])
```

## 원인

`ElasticsearchClient`가 내부적으로 쓰는 `JacksonJsonpMapper`를 인자 없이
(`new JacksonJsonpMapper()`) 생성하고 있었다. 이 생성자는 내부에서 완전히 새로운
순정 Jackson `ObjectMapper`를 만들어 쓰는데, 이 인스턴스는 스프링 부트가 REST API용으로
자동 구성해주는 `ObjectMapper`와는 전혀 별개였다. 스프링 부트가 만들어주는 쪽은
클래스패스에 있는 `jackson-datatype-jsr310`(자바 8 날짜/시간 타입 지원 모듈)을 자동으로
감지해 등록해주지만, ES 클라이언트가 직접 만든 순정 인스턴스에는 이게 빠져 있었다.

<details>
<summary>💡 Jackson은 스프링 부트 것 아닌가?</summary>

Jackson은 스프링팀이 아니라 FasterXML이라는 별도 팀이 만든 완전히 다른 오픈소스
라이브러리다. 다만 스프링 부트가 REST API 응답을 만들 때 이 Jackson을 기본으로
채택해서 `spring-boot-starter-web`을 통해 자동으로 끌고 들어오고, 앱이 시작할 때
"잘 갖춰진 번역기(ObjectMapper) 1개"를 자동으로 만들어 등록해준다. 이때 클래스패스에
`jackson-datatype-jsr310`이 있으면 그 번역기엔 날짜 타입 지원까지 알아서 끼워준다.

문제는 Elasticsearch 클라이언트 라이브러리가 이 스프링 표준 번역기를 가져다 쓰지 않고,
자기 내부에서 `new ObjectMapper()`로 완전히 별개의 번역기를 새로 만들어 쓴다는 점이다.
같은 프로젝트 안에 "스프링이 잘 갖춰놓은 번역기"와 "ES 클라이언트가 혼자 만든 맨몸
번역기"가 공존하는 셈이라, 후자에서만 날짜 타입 직렬화가 실패했다.

</details>

이 예외는 ES로 요청이 나가기도 전에, 요청 본문을 만드는 단계(직렬화)에서 터진
것이었다 — ES는 아무 요청도 받아본 적이 없었다. 기존 단위 테스트는
`ElasticsearchClient`를 Mockito로 mock 처리하고 있어서 실제 직렬화 경로를 전혀
타지 않았고, 그래서 이 버그를 사전에 잡아내지 못했다.

## 해결

`JacksonJsonpMapper` 생성 시, `JavaTimeModule`을 등록한 `ObjectMapper`를 명시적으로
주입하도록 수정했다.

```java
ObjectMapper objectMapper = new ObjectMapper()
    .registerModule(new JavaTimeModule())
    .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
return new Rest5ClientTransport(rest5Client, new JacksonJsonpMapper(objectMapper));
```

재발 방지를 위해 mock이 아니라 testcontainers로 실제 Elasticsearch를 띄워 실제
직렬화 경로를 검증하는 통합 테스트를 추가했다 — 이번 버그가 "mock 기반 테스트로는
원천적으로 잡을 수 없는 종류"였기 때문에, 같은 종류의 재발을 막으려면 실제 인스턴스로
검증하는 테스트가 최소 하나는 있어야 한다고 판단했다.

## 결과

배포 후 실시간 색인과 7일 배치 재조정 둘 다 정상 동작을 확인했다. 배치가 첫 실행되며
그동안 밀려있던 기존 판매중 상품 72건이 한 번에 정상 색인됐다.

## 배운 점

라이브러리가 내부적으로 자기만의 `ObjectMapper`를 새로 만들어 쓰는 경우, 스프링 부트가
자동 구성해준 것과는 완전히 별개의 인스턴스라는 점을 항상 의심해봐야 한다. 같은 프로젝트
안에서도 "날짜 직렬화가 잘 되는 곳"과 "안 되는 곳"이 공존할 수 있다.

또한 로그에서 예외가 안 보인다고 "예외가 없다"고 단정하면 안 된다 — 에러 핸들러가
재시도 중엔 로그를 줄이는 경우가 있으니, 조사 범위를 넓히거나(시간 기준 재조회) 아예
코드 리뷰로 전환하는 판단이 필요했다.
