## 문제

공개/관련 상품 목록 조회(`/browse`)가 데이터 유무와 무관하게 항상 500을 반환했다. 응답은 `SYS001`(서버 내부 오류), 서버 로그엔 `SQLGrammarException`.

## 원인

두 쿼리(`findPublicProducts`, `findRelatedProducts`)가 salesCount를 family 합산으로 구하는 상관 서브쿼리에서 `coalesce(p.parentId, p.id)`로 `parentId`를 참조하는데, 정작 `GROUP BY`에는 `parentId`가 빠져 있었다. PostgreSQL은 이를 거부한다:

```
ERROR: subquery uses ungrouped column "parent_id" from outer query  (SQLState 42803)
```

두 겹의 원인이 겹쳤다:

1. 쿼리 자체가 서브쿼리에서 쓰는 컬럼을 GROUP BY에 안 넣음.
2. 배포 DB의 `product` 테이블에 `id` PRIMARY KEY 제약이 없어, PostgreSQL의 "PK 함수종속" 최적화(PK가 GROUP BY에 있으면 같은 테이블 컬럼 허용)가 안 걸렸다. `ddl-auto: validate`는 컬럼·타입만 보고 PK 제약은 검증하지 않아 그동안 드러나지 않았다.

CI가 못 잡은 이유: 테스트가 H2(`MODE=PostgreSQL`)로 도는데, H2는 이 GROUP BY 함수종속 검사를 강제하지 않아 버그 있는 코드에서도 통과했다.

<details>
<summary><b>🔍 CI(테스트)는 왜 이걸 못 걸렀나</b></summary>

테스트는 운영과 같은 PostgreSQL이 아니라 **H2(인메모리 DB)** 로 돈다. H2는 `MODE=PostgreSQL`로 "PostgreSQL 흉내"를 내지만 문법 호환 수준이고, PostgreSQL의 엄격한 GROUP BY 함수종속 검사(42803)까지는 강제하지 않는다.

- 테스트가 H2에게 "이 쿼리 돼?" → H2: "돼" → 테스트 통과(green) → CI 통과
- 운영은 진짜 PostgreSQL → 같은 쿼리 → 42803 → 500

테스트는 진짜 DB와 대화한 적이 없어서 "안 돼"를 들을 기회가 없었다. "테스트 DB ≠ 운영 DB" 괴리가 만든 사각지대다.

</details>

## 해결

두 쿼리의 `GROUP BY`에 `p.parentId`를 추가했다(`p.id`가 이미 있어 결과 카디널리티 불변, PK 유무와 무관하게 유효한 쿼리가 됨).

실제 `postgres:18.4-alpine`(운영과 동일 버전)로 재현·검증했다: `id`에 PK 제약이 없을 때 수정 전 쿼리는 42803, 수정 후 쿼리는 정상 반환.

## 결과

수정 PR 생성. merge·재배포 후 상품 목록 조회 정상화. 데이터 변경은 없었다.

## 배운 점

- **테스트 DB ≠ 운영 DB.** H2의 PostgreSQL 호환 모드는 문법만 흉내낼 뿐, GROUP BY 함수종속 같은 엄격한 검사는 재현하지 못한다. DB 고유 동작에 의존하는 쿼리는 실 DB 기반 테스트가 필요하다.
- **재현 먼저.** 처음엔 `id`를 PK로 만들어 재현이 안 됐다. PK 유무라는 변수를 찾고 나서야 실패를 재현했고, 그제서야 fix 검증이 유효해졌다.
