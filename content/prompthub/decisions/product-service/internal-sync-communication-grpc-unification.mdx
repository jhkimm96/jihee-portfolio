## 배경

order-service가 product-service의 내부 조회(주문 스냅샷, 카트 스냅샷, 콘텐츠 조회)를 호출하는
경로가 REST(Feign)와 gRPC 두 가지로 동시에 존재했다. 같은 계약이 두 곳에서 유지되면서 한쪽만
바뀌었을 때 조용히 어긋날 위험이 있었다.

## 고려한 선택지

- **REST(Feign) 유지**: 익숙하고 curl 등으로 디버깅하기 쉽다. 대신 계약을 두 곳에서 유지하는
  비용과, 한쪽만 바뀌었을 때 몰래 어긋나는 위험이 계속 남는다.
- **gRPC로 단일화**: `.proto` 하나로 계약이 강제돼 타입 안전성이 생기고, HTTP/2 멀티플렉싱과
  protobuf 바이너리 직렬화로 지연·페이로드가 줄어든다. 대신 사람이 직접 호출/디버깅하기
  어려워지고 계약 변경 규칙(필드 번호 재사용 금지 등)을 더 엄격히 지켜야 한다.

## 결정

내부 서비스 간 동기 호출은 gRPC로 단일화한다. product-service의 `/internal/products/**`
REST 엔드포인트(주문 스냅샷·카트 스냅샷·콘텐츠 조회)는 order-service의 gRPC 전환이 끝나는
대로 제거한다.

## 결과

- HTTP/2 커넥션 재사용과 protobuf 직렬화로 커넥션 오버헤드·페이로드 크기가 줄었다.
- `.proto` 계약이 컴파일 타임에 강제되어, REST/gRPC 응답이 서로 몰래 어긋나는 위험이 사라진다.
- 전환 기간 동안 REST 경로가 아직 코드에 남아 있다 — 2026-07-20 품질 스냅샷에서 미사용 코드로
  확인됨. gRPC 전환이 끝나는 대로 정리한다.
- gRPC는 REST보다 사람이 직접 호출해 확인하기 어렵다는 트레이드오프를 가져간다.
