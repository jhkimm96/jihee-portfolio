## 배경

PromptHub는 원래 EC2의 공인 IP/DNS로 접근하고 있었고, 아직 자체 도메인이나 신뢰
가능한 TLS 인증서를 준비하지 못한 상태였다. Kubernetes로 전환하면서 외부 진입점
(Ingress)을 HTTP(80)로 열지, 처음부터 HTTPS(443)까지 갖출지 선택해야 했다.

## 고려한 선택지

- **자체 도메인 + TLS 인증서를 먼저 준비하고 HTTPS로 시작**: 처음부터 안전한
  채널을 제공한다. 브라우저의 "혼합 콘텐츠(mixed content)" 정책 문제도 없다
  (HTTPS 프론트엔드에서 HTTP API를 직접 호출하면 브라우저가 차단하는 문제). 하지만
  도메인 구매·DNS 설정·인증서 발급 및 자동 갱신까지 준비 작업이 추가로 필요해서,
  Kubernetes 전환 자체의 검증이 늦어진다.
- **일단 EC2 공인 DNS + HTTP 80만으로 시작**: 도메인·인증서 없이 바로 클러스터
  통신과 cutover 전 스모크 테스트가 가능하다. 대신 브라우저 혼합 콘텐츠 문제
  때문에, HTTPS로 서빙되는 프론트엔드(예: Vercel)에서 이 HTTP API를 직접 호출하면
  차단될 수 있다 — 프론트엔드 쪽에서 서버사이드 프록시·리라이트를 거치거나,
  도메인·HTTPS가 갖춰질 때까지 우회가 필요하다.

## 결정

첫 단계는 EC2 공인 DNS와 HTTP 80만 외부 진입점으로 쓴다. 자체 도메인과 신뢰
가능한 TLS 인증서가 준비되기 전에는 443을 Kubernetes 진입점으로 열지 않는다.
보안 그룹에는 443 규칙 자체는 남겨두되(향후 확장을 위해), 실제로 Kubernetes
Ingress가 443을 받도록 설정하는 건 도메인·인증서 준비 이후로 미룬다.

## 결과

- Kubernetes 전환 자체의 검증(클러스터 통신, cutover 스모크 테스트)을 도메인·
  인증서 준비와 분리해서 먼저 끝낼 수 있었다.
- 대신 HTTPS로 서빙되는 프론트엔드에서 이 API를 직접 호출하면 브라우저 혼합
  콘텐츠 정책에 막힌다 — 실제 프런트엔드 연동 전에는 자체 도메인+HTTPS 구성이나
  프론트엔드 쪽 프록시가 반드시 필요하다.
- 이건 "먼저 인프라 전환 자체를 검증하고, HTTPS는 별도 단계로 뒤에 붙인다"는
  의도된 순서이지, 보안을 영구히 포기한 게 아니다.
