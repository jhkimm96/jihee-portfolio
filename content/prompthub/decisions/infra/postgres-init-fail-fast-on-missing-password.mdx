## 배경

PostgreSQL StatefulSet이 처음 뜰 때, 초기화 스크립트(ConfigMap)가 서비스별 DB 계정
(user-service, product-service 등)의 비밀번호를 환경변수로 받아서 설정한다. 근데
이 환경변수 중 하나라도 누락되거나 빈 값이면, 스크립트가 **예측 가능한 기본
비밀번호**로 조용히 대체해서 계속 진행하던 문제가 있었다. 즉 Secret 설정 실수가
있어도 초기화는 "조용히 성공"하고, 대신 아무도 모르는 약한 고정 비밀번호로 DB
계정이 만들어졌다.

## 고려한 선택지

- **지금처럼 기본값으로 대체하고 계속 진행**: 초기화가 절대 실패하지 않아서
  배포가 매끄럽다. 하지만 Secret 설정 실수를 발견할 방법이 없고, 약한 비밀번호로
  DB 계정이 만들어질 수 있다 — 보안 구멍이 조용히 생긴다.
- **필수 환경변수가 없으면 초기화 자체를 실패시킨다**: 문제를 즉시 드러낸다
  (배포가 눈에 띄게 실패하니 바로 알아챌 수 있음). 대신 Secret 준비가 안 됐으면
  배포가 막힌다 — 그런데 이건 오히려 의도한 동작이다. 잘못된 상태로 조용히
  넘어가는 것보다 낫다.

## 결정

초기화 스크립트에 `set -euo pipefail`을 적용하고, `psql` 실행 전에 서비스별
비밀번호 5개(user·product·order·payment·settlement) 각각의 존재·빈 값 여부를
검증한다. 하나라도 없거나 비어있으면 그 자리에서 종료하고 넘어가지 않는다.
예측 가능한 기본값 fallback은 완전히 제거했다. 이 변경은 Kubernetes 쪽
ConfigMap에만 적용했고, 기존 `docker-compose.yml`과 로컬 초기화 스크립트는
손대지 않았다.

## 결과

- Secret 설정 실수는 이제 "초기화 실패"로 바로 드러난다 — 약한 기본 비밀번호로
  조용히 DB 계정이 만들어지는 경로가 완전히 막혔다.
- 이미 예전에 이 방식으로 초기화된 DB 계정의 비밀번호는 이번 변경으로 자동
  교체(회전)되지 않는다 — 이 변경은 앞으로의 새 초기화만 막는다.
- Docker Compose 경로는 이번 변경 범위 밖이라, 그쪽은 여전히 예전 동작(기본값
  대체)을 쓴다 — Kubernetes cutover가 완료되면 정리 대상이다.
