## 배경

Kubernetes 자체적으로 서비스 검색(Service + DNS)과 설정 관리(ConfigMap) 기능을
갖고 있다. 근데 PromptHub는 이미 Spring Cloud 생태계에서 Eureka(서비스 등록·발견)와
Config Server(중앙 설정 서버)를 쓰고 있었다. Kubernetes로 전환하면서 이 둘을
Kubernetes 네이티브 기능으로 바로 교체할지, 아니면 유지할지 선택해야 했다.

## 고려한 선택지

- **Kubernetes 네이티브로 즉시 전환**: 서비스 검색은 Kubernetes Service DNS로,
  설정은 ConfigMap으로 완전히 대체한다. 장기적으로 Spring Cloud 의존성을 줄이고
  Kubernetes 표준 방식에 가까워진다. 하지만 API Gateway의 `lb://SERVICE-NAME`
  라우팅 로직, 각 서비스의 Eureka 클라이언트 연동, Config Server를 바라보는
  bootstrap 설정까지 애플리케이션 코드·설정을 동시에 대폭 바꿔야 한다.
- **Eureka·Config Server 유지**: 애플리케이션 쪽 코드는 그대로 두고, 그 서비스들
  자체를 Kubernetes Pod으로 올리기만 한다. Kubernetes 전환이라는 변수 하나만
  다루게 되어 각 변경의 영향을 검증하기 쉽다. 대신 Kubernetes가 이미 제공하는
  기능과 겹치는 컴포넌트를 그대로 안고 간다.

## 결정

"Kubernetes 전환과 애플리케이션 구조 변경을 동시에 진행하지 않는다"는 원칙에 따라,
Eureka 기반 서비스 등록·`lb://SERVICE-NAME` 라우팅과 Spring Cloud Config Server를
그대로 유지했다. 둘 다 Kubernetes 위에서 하나의 Deployment로 올라간다. Kubernetes
Service DNS는 Pod 간 기본 주소 확인(`postgres:5432`, `kafka:9092`, `config:8888`
등)에만 쓰고, HTTP 서비스 등록·조회는 여전히 Eureka가 담당한다.

## 결과

- 변수를 하나로 좁혀서 "이게 Kubernetes 전환 때문에 생긴 문제인지, 애플리케이션
  구조를 바꿔서 생긴 문제인지" 헷갈릴 일이 없어졌다.
- 대신 Eureka와 Config Server라는 별도 컴포넌트를 계속 운영해야 한다 — Kubernetes가
  이미 제공하는 기능과 중복되는 부분이 있다. Eureka가 죽으면 신규 등록·조회가
  실패하고 Gateway 라우팅이 불안정해지는 등, Kubernetes Service만 썼다면 없었을
  장애 지점이 하나 더 생겼다.
- 이건 "지금은 유지, 나중에 Kubernetes 네이티브 전환은 별도 이슈에서 재검토"하는
  의도된 단계적 접근이다.
