## 배경

FE는 기본적으로 로컬(localhost:3000)에서 AWS에 배포된 API Gateway를 보며 개발한다. mock 없이 실제 AWS 백엔드와 DB를 공용으로 쓰는 게 기본 워크플로우다. 다만 특정 서비스(예: product-service)의 백엔드 코드를 수정 중일 때는, 그 서비스만 로컬로 띄우고 나머지(설정·디스커버리·게이트웨이 포함)는 AWS 배포본을 그대로 쓰는 하이브리드 테스트가 필요했다. 매번 서비스 전체를 로컬에 띄우지 않고도, 로컬 프론트 + 로컬 서비스 1개 + 원격 나머지 조합으로 실제 플로우를 확인하기 위함이다.

## 고려한 선택지

- **Eureka 자체 등록**: 로컬에서 띄운 서비스를 AWS Eureka에 그대로 등록해 게이트웨이가 자연스럽게 호출하게 만드는 방법. AWS 인프라를 확인해보니 외부에 노출된 포트가 게이트웨이뿐이고, 나머지 서비스는 EC2 로컬호스트 바인딩이거나 내부 브리지 네트워크에서만 통신 가능해서 AWS에서 개발자 로컬 PC로 역방향 접속할 경로 자체가 없었다. 터널/VPN을 새로 구축하지 않는 한 불가능하다. 설령 가능해도 같은 서비스명으로 AWS 인스턴스와 함께 등록되면 랜덤 분산으로 "내 로컬 코드가 아니라 AWS 인스턴스가 응답"하는 비결정적 상황이 생기고, 같은 dev 백엔드를 쓰는 다른 팀원 요청에도 영향을 준다.
- **서비스별 axios 인스턴스 + 삼항연산자(기각)**: Next.js rewrite 없이 클라이언트 코드에서 바로 목적지를 나누는 방법. AWS로 가는 나머지 트래픽은 지금 방식과 동일하게 게이트웨이를 그대로 거치지만, 로컬로 띄운 서비스 하나로 가는 요청만큼은 브라우저가 게이트웨이를 건너뛰고 직접 호출하게 되어 CORS 문제가 생긴다.

  ```
  [기각] 로컬 서비스 하나로 가는 요청 — 브라우저가 직접 호출
  브라우저(localhost:3000) ──cross-origin──▶ 로컬 product-service(localhost:8082)
                                              └─ CORS 사전 설정 없으면 브라우저가 응답을 막음

  [채택] 같은 요청 — Next.js가 대신 중계
  브라우저(localhost:3000) ──same-origin──▶ Next.js dev server(localhost:3000)
                                              └─서버 대 서버(CORS 규정 미적용)──▶ 로컬 product-service(localhost:8082)
  ```

  이를 해결하려면 로컬로 띄운 백엔드 서비스마다 CORS 허용 설정을 추가해야 하는데, 이는 백엔드 소스를 건드리는 일이라 "FE/배포 설정만으로 해결한다"는 제약과 맞지 않아 기각했다.
- **서비스별 FE 라우팅(채택)**: 방향을 뒤집어 FE(브라우저/Next.js dev server)가 서비스별로 요청 목적지를 나눠 보내는 방식. 개발자의 로컬 PC는 AWS도 자기 자신(localhost)도 문제없이 호출할 수 있어 새 인프라가 필요 없다.

## 결정

처음에는 서비스별(env 플래그 이름 + 대상 경로 prefix + 기본 포트)을 하나의 설정 배열로 만들어 5개 서비스를 모두 등록해뒀다. 하지만 이는 API 게이트웨이의 라우트 테이블을 FE 코드에 그대로 복제해 유지보수해야 하는 문제가 있었다 — 백엔드가 새 엔드포인트를 추가할 때마다 이 배열도 같이 고쳐야 했다. 그래서 공용 배열을 없애고, 지금 로컬로 띄운 서비스 하나의 경로 목록(`NEXT_PUBLIC_LOCAL_PROXY_PATHS`)과 대상(`NEXT_PUBLIC_LOCAL_PROXY_TARGET`)만 개인 `.env.local`에 적는 단일 슬롯 방식으로 단순화했다. 이 두 값은 커밋되지 않는 개인 설정이라, 어떤 경로가 어느 서비스 것인지는 지금 그 서비스를 로컬로 띄운 사람만 알면 된다. Next.js rewrite 생성과 인증 헤더 주입(게이트웨이가 하던 X-User-Id/X-User-Role 주입을 로그인 유저 정보로 대신)은 이 값을 읽는 함수 하나(`getLocalProxyConfig`)를 공유한다. 프로덕션 환경이면 값과 무관하게 항상 비활성화되는 이중 안전장치는 그대로 유지했다.

## 결과

기존에 정산·결제 서비스에만 있던 개별 프록시 방식(전용 경로 + 고정 테스트 유저 ID)도 이 방식으로 통합됐고, 이후 경로 목록을 아예 없애는 방향으로 한 번 더 단순화했다. MSW mock 레이어의 passthrough 로직도 같은 단일 설정을 공유하도록 맞춰, 경로 목록이 여러 곳에서 따로 관리되며 어긋나는 문제를 근본적으로 없앴다(이후 MSW 자체를 전부 제거하면서 이 부분은 자연히 사라졌다). 단위 테스트로 라우팅 규칙 생성과 헤더 주입 로직을 검증했고, gRPC 터널이 필요한 조회나 Kafka 로컬 테스트는 이번 스코프에 포함하지 않고 알려진 제약으로 남겼다.
